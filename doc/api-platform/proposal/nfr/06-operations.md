## §6.0 前提と背景

### §6.0.1 用語整理

| 用語 | 定義 |
|---|---|
| **オンコール** | 障害発生時の呼び出し対応体制 |
| **SLO / Error Budget** | サービスレベル目標と、それを達成するための「壊してよい予算」 |
| **Patch management** | OS / ライブラリ / コンテナイメージのパッチ適用 |

### §6.0.2 なぜここ（§6）で決めるか

運用性は **「低運用負荷・コスト」基本方針の中核**。マネージドサービスを活用しつつ、デプロイ・パッチ・監視・障害対応の標準を本章で定義する。

### §6.0.3 IPA グレード対応

| IPA 中項目 | 本章での対応 |
|---|---|
| 通常運用 / 障害時運用 | §6.1 / §6.3 |
| 監視 | §6.1 |
| 保守運用（パッチ） | §6.2 |
| 運用体制 | §6.4 |

### §6.0.4 §6.0.A 本標準のスタンス

| 基本方針 | 本章での具体化 |
|---|---|
| 絶対安全 | アラート設計の標準（誤検知抑制、エスカレーション） |
| どんなアプリでも | アプリ別のオンコール体制テンプレ |
| 効率よく | デプロイは IaC + CI/CD 必須、手作業禁止 |
| 運用負荷・コスト最小 | マネージドサービスのアップデート追従、自前運用は最小限 |

## §6.1 監視・アラート

**このサブセクションで定めること**：必須メトリクス・ログ・アラートの標準。
**主な判断軸**：SLO 連動、誤検知抑制。
**§6 全体との関係**：§FR-8 観測性の運用面。

### §6.1.1 ベースライン

- **必須メトリクス**：§FR-API-8 §8.3 の標準セット
- **ダッシュボード**：CloudWatch Dashboards / QuickSight のアプリ別テンプレ
- **アラート**：
  - **重大（Severity 1）**：オンコール電話 / PagerDuty
  - **警告（Severity 2）**：Slack / メール
  - **情報（Severity 3）**：通知のみ
- **アラート Runbook**：各アラートに対応する手順書を必須（Confluence / Notion / README）

## §6.2 デプロイ・パッチ

**このサブセクションで定めること**：デプロイ方式と OS / ライブラリパッチの標準。
**主な判断軸**：手作業禁止、ロールバック容易、Critical は段階デプロイ。
**§6 全体との関係**：§FR-6 ECS デプロイ・§FR-5 Lambda デプロイ。

### §6.2.1 ベースライン

#### デプロイ

- **IaC（CDK / Terraform / SAM）必須**、手作業禁止
- **CI/CD パイプライン**（CodePipeline / GitHub Actions）
- **Critical Tier は段階デプロイ**（Canary / Blue-Green / Linear）
- **ロールバック手段必須**（previous version への即時切り戻し）

#### パッチ

- **Lambda ランタイム**：自動アップデート（マネージド）、廃止予定（Deprecation）を 60 日前にチェック
- **ECS / Fargate コンテナ**：ベースイメージのバージョン管理、月次再ビルド
- **Amazon Linux 2023 / Amazon Linux Container Image** を既定
- **依存ライブラリ**：SBOM 生成、Dependabot 等で更新追従

- Q: コンテナベースイメージの **再ビルド頻度**（月次 / 週次）→ `API-C-1412`

## §6.3 障害対応・インシデント

**このサブセクションで定めること**：障害発生時の手順・コミュニケーション・ポストモーテム。
**主な判断軸**：影響可視化、再発防止、組織学習。
**§6 全体との関係**：§NFR-1 可用性 SLO 違反時の対応。

### §6.3.1 ベースライン

- **検知**：CloudWatch Alarm → PagerDuty → オンコール
- **エスカレーション基準**：Severity ごとの応答時間 SLA
- **ステータスページ**：顧客向け（Statuspage / Atlassian Statuspage / 自社）
- **ポストモーテム**：blameless、Confluence / Notion で公開
- **AWS Health Dashboard**：AWS 側障害の Subscribe

### §6.3.2 TBD / 要確認

- Q: ポストモーテム **公開範囲**（社内全部 / チーム内）→ `API-C-1422`

## §6.4 体制・SLA

**このサブセクションで定めること**：運用体制と社内 SLA の標準。
**主な判断軸**：規模と Tier に応じたチーム構成。
**§6 全体との関係**：§NFR-1 SLO 達成の組織前提。

### §6.4.1 ベースライン

| 項目 | Critical | Standard | Internal |
|---|---|---|---|
| オンコール | 24/7 | 業務時間 + 緊急 | 業務時間 |
| 初動応答 SLA | < 15 分 | < 1 時間 | < 4 時間 |
| ポストモーテム | 必須 | 重大時のみ | 任意 |

- **AWS サポート契約**：本番アカウントは Business / Enterprise 推奨
- **DevOps Guru / Trusted Advisor** のアラート受信
