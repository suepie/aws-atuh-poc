## §9.0 前提と背景

### §9.0.1 用語整理

| 用語 | 定義 |
|---|---|
| **バージョニング** | API の互換性破壊変更を識別する仕組み（URL パス / Header / Query） |
| **後方互換** | 旧バージョンクライアントが動作し続ける性質 |
| **Deprecation** | 廃止予告、利用者に移行期間を与える |
| **Breaking change** | 互換性を破る変更 |

### §9.0.2 なぜここ（§9）で決めるか

API は **複数の利用者（社内アプリ・外部 Partner）が依存する契約**。本標準で互換性ルールを定めることで、利用者を混乱させない更新サイクルを保証する。

### §9.0.4 §9.0.A 本標準のスタンス

| 基本方針 | 本章での具体化 |
|---|---|
| 絶対安全 | Breaking change は必ず deprecation 期間を経る、突然停止禁止 |
| どんなアプリでも | URL path バージョニング（/v1, /v2）を既定 |
| 効率よく | 同時 N バージョン稼働の上限を明文化 |
| 運用負荷・コスト最小 | 旧バージョン稼働コストは利用者に明示、移行ロードマップ提示 |

## §9.1 API バージョニング規約

**このサブセクションで定めること**：バージョン表記・宣言・廃止予告の標準。
**主な判断軸**：利用者から見て分かりやすい、ツール対応の容易性。
**§9 全体との関係**：§9.2 後方互換のキャリア。

### §9.1.1 ベースライン

- **URL path 方式**：`/v1/...`、`/v2/...`
  - 採用理由：明示的、ツール対応容易、API Gateway Stage マッピングと相性良
- **代替案**：Header 方式（`Accept: application/vnd.example.v2+json`）— 採用は要件次第
- **OpenAPI 仕様書**：必須、`info.version` を明示、Service Catalog で公開
- **Deprecation**：HTTP ヘッダ `Deprecation: true` + `Sunset: <date>`、利用者通知

## §9.2 後方互換性ルール

**このサブセクションで定めること**：Breaking change の定義と移行プロセス。
**主な判断軸**：利用者保護、運用負荷とのバランス。
**§9 全体との関係**：§9.1 規約の運用ルール。

### §9.2.1 ベースライン

#### Breaking change の例

- フィールド削除・型変更
- 必須フィールド追加
- ステータスコードの意味変更
- URL 構造変更

#### Non-breaking change（互換維持）

- フィールド追加（オプショナル）
- ステータスコード追加（既存パスは維持）
- 内部実装変更

#### Deprecation 期間

| API カテゴリ | 旧バージョン維持期間 |
|---|---|
| **Public** | 12 ヶ月 |
| **Partner** | 個別契約（最低 6 ヶ月） |
| **Internal** | 3 ヶ月 |
| **Private** | 個別 |

#### 同時稼働バージョン数

- **最大 2 バージョン**（最新 + 1 個前）を標準とする
- 例外は申請制

## §9.3 既存アプリの本標準への移行

**このサブセクションで定めること**：既存アプリを本標準に適合させる手順とロードマップ。
**主な判断軸**：影響度・コスト・優先度。
**§9 全体との関係**：本標準の現実的な普及戦略。

### §9.3.1 ベースライン

#### 移行 Tier

| Tier | 対応 | 期限 |
|---|---|---|
| **新規アプリ** | 本標準必須 | 即時 |
| **既存 Critical アプリ** | 計画的移行 | 12 ヶ月 |
| **既存 Standard アプリ** | 通常リファクタリングサイクルで適用 | 24 ヶ月 |
| **既存 Internal アプリ** | 任意 / 老朽化更新時 | - |
| **レガシー（廃止予定）** | 適用免除 | - |

#### 移行手順テンプレ

1. 現状評価（公開境界 / 認証 / 流量 / 観測性のチェック）
2. ギャップ分析（本標準との差分）
3. 段階移行プラン（公開境界 → 認証 → 観測性 → ガードレール）
4. 移行後の継続評価（Config Rule）
