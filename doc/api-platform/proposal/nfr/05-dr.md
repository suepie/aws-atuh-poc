## §5.0 前提と背景

### §5.0.1 用語整理

| 用語 | 定義 |
|---|---|
| **DR (Disaster Recovery)** | 災害復旧。リージョン障害等の大規模障害からの復旧 |
| **BCP (Business Continuity)** | 事業継続計画 |
| **RPO (Recovery Point Objective)** | 復旧時の許容データ損失時間 |
| **RTO (Recovery Time Objective)** | 復旧目標時間 |
| **Active-Active / Active-Standby** | 複数リージョンを同時利用 / 待機系として保持 |

### §5.0.2 なぜここ（§5）で決めるか

リージョン障害は単一マネージドサービスの SLA では救えない（§NFR-1 の延長線にない）。本章で **どのカテゴリの API にリージョン障害対策まで必要か**を切り分ける。

### §5.0.3 IPA グレード対応

| IPA 中項目 | 本章での対応 |
|---|---|
| 災害対策 / 復旧水準 | §5.1 / §5.2 |
| 復旧範囲 / 業務継続性 | §5.2 |

### §5.0.4 §5.0.A 本標準のスタンス

| 基本方針 | 本章での具体化 |
|---|---|
| 絶対安全 | Critical Tier はマルチリージョン待機系 + データ複製を標準化 |
| どんなアプリでも | Tier 別 RPO/RTO テンプレで網羅 |
| 効率よく | Active-Standby を既定、Active-Active は限定（コスト高） |
| 運用負荷・コスト最小 | DR テスト・切替訓練の手順は IaC で自動化 |

## §5.1 API カテゴリ別 RPO/RTO

**このサブセクションで定めること**：Tier 別に災害復旧目標を定義。
**主な判断軸**：業務影響度、コスト許容度。
**§5 全体との関係**：§5.2 構成パターンの設計目標。

### §5.1.1 ベースライン

| Tier | RPO | RTO | 構成例 |
|---|---:|---:|---|
| **Critical** | < 5 分 | < 30 分 | Active-Active マルチリージョン + DynamoDB Global Tables / Aurora Global Database |
| **Standard** | < 1 時間 | < 4 時間 | Active-Standby マルチリージョン |
| **Internal** | < 24 時間 | < 24 時間 | 単一リージョン（リージョン障害は受容） |
| **Batch / Async** | < 24 時間 | < 48 時間 | 単一リージョン |

## §5.2 マルチリージョン構成パターン

**このサブセクションで定めること**：DR 構成の標準パターン。
**主な判断軸**：マネージドサービスのリージョン間サポート、データ複製の選択肢。
**§5 全体との関係**：§5.1 RPO/RTO 達成手段。

### §5.2.1 ベースライン

| パターン | 構成 | 適用 Tier |
|---|---|---|
| **Active-Active** | Route53 健康チェック + 重み付け / Latency-based、DynamoDB Global Tables / Aurora Global Database | Critical |
| **Active-Standby（Warm）** | Route53 failover、待機系を最小規模で常時稼働、データ複製は cross-region replication | Standard |
| **Active-Standby（Pilot Light）** | データのみ複製、コンピュートはオフ、災害時に scale-up | Standard 低コスト要件時 |
| **Backup & Restore** | バックアップのみ、復旧時にゼロから構築 | Internal / Batch |

- **データ複製の選択肢**：
  - DynamoDB Global Tables（Active-Active 自動）
  - Aurora Global Database（< 1 sec replication）
  - S3 Cross-Region Replication
  - RDS Read Replica（cross-region）
- **API Gateway / Lambda** はリージョン別デプロイ、IaC で同期

## §5.3 DR テスト・切替訓練

**このサブセクションで定めること**：DR 構成の継続検証。
**主な判断軸**：実効性、業務影響なし、自動化。
**§5 全体との関係**：§5.2 の保証手段。

### §5.3.1 ベースライン

- **半年 1 回**：DR 切替訓練（業務時間外）
- **AWS Resilience Hub** で構成評価
- **Route53 failover** のドライラン
- **IaC ベース**で復旧手順を自動化
