## §8.0 前提と背景

### §8.0.1 用語整理

| 用語 | 定義 |
|---|---|
| **CUR** | Cost and Usage Report、AWS の利用料・使用量詳細レポート |
| **Cost allocation tag** | リソースに付与するタグで CUR / Cost Explorer に表示 |
| **Savings Plan / RI** | コミットメント割引 |
| **按分** | 共有リソースのコストを利用量比で割り振り |

### §8.0.2 なぜここ（§8）で決めるか

「**運用負荷・コスト最小**」基本方針の中核。コスト可視化なしに按分・予算管理はできない。本章は §FR-4 課金按分の **NFR 視点**（コスト目標・予算・通知）を担う。

### §8.0.3 §8.0.A 本標準のスタンス

| 基本方針 | 本章での具体化 |
|---|---|
| 絶対安全 | コスト暴騰検知（AWS Budgets / Anomaly Detection）必須 |
| どんなアプリでも | アプリ別ダッシュボード標準、按分パイプライン共通 |
| 効率よく | マネージドサービス（CUR / Cost Explorer / QuickSight）優先 |
| 運用負荷・コスト最小 | Savings Plan / Compute SP の **全社合算最適化**、各アプリでの個別購入禁止 |

## §8.1 コストベースライン

**このサブセクションで定めること**：標準アーキ単位のコスト指標。
**主な判断軸**：継続観測、比較可能性。
**§8 全体との関係**：§FR-4 課金按分の指標化。

### §8.1.1 ベースライン

- **CUR 2.0（FOCUS-aligned）**：Org 管理アカウントから S3 出力、Athena で集計
- **アプリ別ダッシュボード**：QuickSight 標準テンプレで提供
- **指標例**：
  - Request あたりコスト（USD / 1M req）
  - テナント別月額コスト
  - 環境別（prod / stg / dev）構成比
  - 共有リソース（VPC・データ転送）の按分済みコスト

- Q: コスト指標の **目標値**（USD / 1M req の上限等）→ `API-D-1602`

## §8.2 予算管理・アラート

**このサブセクションで定めること**：予算超過の予防的検知。
**主な判断軸**：早期検知、誤検知抑制。
**§8 全体との関係**：§FR-3 流量制御がコスト上限を守るが、最終防衛線として予算アラート。

### §8.2.1 ベースライン

- **AWS Budgets**：アカウント別・サービス別・タグ別の予算設定
  - 80% 到達で通知（Slack / メール）
  - 100% 到達でエスカレーション
- **AWS Cost Anomaly Detection**：異常検知（ML ベース、無料）必須有効化
- **AWS WAF 使用量アラート**：Bot Control / Shield Advanced のコスト監視

### §8.2.2 TBD / 要確認

- Q: 予算の **既定アラート閾値**（50% / 80% / 100% / 120%）→ `API-C-1611`
- Q: 異常検知の **通知先** → `API-C-1612`

## §8.3 コスト最適化

**このサブセクションで定めること**：継続的コスト最適化の手段標準。
**主な判断軸**：マネージド最適化、運用負荷を増やさない。
**§8 全体との関係**：§FR-5/6 実装ランタイムの選定にも影響。

### §8.3.1 ベースライン

- **Compute Savings Plan**：全社合算で最適購入（個別アプリでの購入禁止）
- **arm64 (Graviton) 推奨**：Lambda / ECS で 20% 安
- **Spot タスク**：バッチ・stateless ECS タスクで採用
- **Aurora Serverless v2 min ACU=0**（2025、MySQL 3.08+）
- **CloudWatch Logs Retention 必須**：無期限保管禁止
- **不要 NAT Gateway / VPC Endpoint の整理**（データ転送費削減）

- Q: Savings Plan の **コミット率**（70% / 80%）→ `API-D-1621`
- Q: arm64 移行の **既存アプリ計画** → `API-D-1622`
