# 要件定義ヒアリング戦略

> 最終更新: 2026-04-21
> 目的: PoC 結果を踏まえた共有認証基盤の要件定義において、誰に・何を・どの順序で確認するかの計画

---

## 1. ヒアリングの全体方針

### 1.1 基本戦略

```
Phase A: 事業要件の確認（ビジネス側）
    ↓
Phase B: 技術要件の確認（利用システム側）
    ↓
Phase C: 運用・セキュリティ要件の確認（インフラ・セキュリティ側）
    ↓
Phase D: 最終判断材料の確定（意思決定者）
```

### 1.2 進め方の原則

1. **PoC 結果を先に共有してからヒアリング**する（知識の非対称性を解消）
2. **定量データを先に提示**して、判断を求める（「どう思いますか」ではなく「A と B どちらですか」）
3. **未決事項リストを事前に配布**して、ヒアリング前に各自の回答を準備してもらう
4. **議事録は即日共有**して、認識の齟齬を早期に検出する

## 2. ステークホルダー別ヒアリング計画

### 2.1 Phase A: 事業要件の確認

**対象**: プロダクトオーナー / 事業企画 / 営業

| # | 確認事項 | 背景（なぜ聞くのか） | 期待する回答形式 |
|---|---------|--------------------|--------------------|
| A-1 | 対象システム数と優先順位 | 共有基盤の初期スコープ確定 | システムリスト + 優先度 |
| A-2 | 想定顧客数・MAU 規模 | コスト試算の入力値（損益分岐 175,000 MAU） | 1 年後 / 3 年後の MAU 予測 |
| A-3 | 顧客の IdP 種類 | Entra ID / Okta / LDAP / SAML の分布 | 既知顧客 + 想定分布 |
| A-4 | サービス展開リージョン | DR 設計・データ所在地の制約 | 国内のみ / グローバル |
| A-5 | 初回リリース時期の目標 | スケジュール制約 | 目標日 |
| A-6 | 新規顧客追加の頻度 | オンボーディングフローの設計粒度 | 月 N 件程度 |
| A-7 | 既存認証からの移行有無 | 移行計画の必要性 | 移行元システム・ユーザー数 |
| A-8 | ブランディング要件 | ログイン画面のカスタマイズ範囲 | ロゴ・色・ドメイン名 |

**事前配布資料**:
- PoC 総括（本ドキュメントの要約版）
- Cognito vs Keycloak コスト比較表（ADR-006 ベース）
- Identity Broker パターン概要図

### 2.2 Phase B: 技術要件の確認

**対象**: 利用システムの開発チーム / テックリード

| # | 確認事項 | 背景 | 期待する回答形式 |
|---|---------|------|-----------------|
| B-1 | 各システムの技術スタック | バックエンドの JWT ライブラリ選定 | 言語・フレームワーク一覧 |
| B-2 | 現在の認証方式 | 移行の難易度評価 | 認証方式・セッション管理方法 |
| B-3 | 必要なユーザー属性 | JWT クレーム設計の入力 | 属性リスト（テナント、ロール、部署等） |
| B-4 | 認可の粒度 | Lambda Authorizer vs バックエンド認可の境界 | ロール / リソース / アクション |
| B-5 | API 間トークンリレーの有無 | マイクロサービス間のトークン伝播設計 | サービス間呼び出しの有無・パターン |
| B-6 | オフライン / M2M 認証の有無 | Client Credentials フローの必要性 | バッチ処理・サーバー間通信の有無 |
| B-7 | フロントエンドの種類 | PKCE フロー以外の対応要否 | SPA / SSR / モバイル / デスクトップ |
| B-8 | トークン有効期限の要件 | Access Token / Refresh Token の TTL 設計 | セキュリティ vs UX のバランス |
| B-9 | ログアウト要件 | SSO ログアウトの完全性 | ローカルのみ / 全システム連動 |
| B-10 | Webhook / イベント通知 | ユーザー作成・削除時の連携 | 必要なイベントリスト |

**事前配布資料**:
- 認可アーキテクチャ設計（authz-architecture-design.md の要約）
- JWT クレーム設計案
- バックエンド実装ガイド（言語別サンプル）

### 2.3 Phase C: 運用・セキュリティ要件の確認

**対象**: インフラチーム / セキュリティチーム / 情シス

| # | 確認事項 | 背景 | 期待する回答形式 |
|---|---------|------|-----------------|
| C-1 | 可用性要件（SLA） | Cognito 99.9% vs Keycloak 自前 HA | 99.9% / 99.95% / 99.99% |
| C-2 | DR 要件（RTO / RPO） | DR 構成のコスト判断 | RTO: N 分、RPO: N 分 |
| C-3 | 監査ログ要件 | ログ保存期間・検索性・改ざん防止 | 保存期間、対象イベント |
| C-4 | コンプライアンス要件 | 個人情報保護法、業界固有規制 | 適用法規リスト |
| C-5 | データ所在地要件 | リージョン選定・データレジデンシー | 国内限定 / 特定リージョン |
| C-6 | ネットワーク要件 | VPN / PrivateLink / インターネット経由 | 接続方式 |
| C-7 | 既存の監視基盤 | CloudWatch / Datadog / Grafana 等 | ツール名・統合方法 |
| C-8 | インシデント対応体制 | 認証障害時の連絡フロー | オンコール体制の有無 |
| C-9 | 変更管理プロセス | IdP 追加・設定変更のリードタイム | 承認フロー・SLA |
| C-10 | セキュリティテスト要件 | ペネトレーションテスト・脆弱性スキャン | 頻度・スコープ |
| C-11 | パスワードポリシー | ローカルユーザーのパスワード要件 | 最小文字数・複雑性・履歴 |
| C-12 | アカウントロック要件 | ブルートフォース対策 | 試行回数・ロック時間 |
| C-13 | MFA 要件 | 全ユーザー必須？条件付き？ | 対象範囲・許可方式 |

**事前配布資料**:
- JWKS 公開判断の根拠（jwks-public-exposure.md 要約）
- DR コスト比較表
- MFA 検証結果サマリー

### 2.4 Phase D: 最終判断材料の確定

**対象**: 意思決定者 / アーキテクト / プロジェクトマネージャー

| # | 確認事項 | 背景 | 判断に必要な情報 |
|---|---------|------|-----------------|
| D-1 | Cognito vs Keycloak の最終判断 | PoC 結果に基づく選定 | Phase A-C の結果を統合 |
| D-2 | 段階的移行 vs ビッグバン | リスクとスケジュールのバランス | 移行戦略の選定 |
| D-3 | 運用体制の確定 | 専任チーム / 兼任 / 外部委託 | 人員・スキルセット |
| D-4 | 予算の確定 | インフラ + 開発 + 運用コスト | 年間予算枠 |
| D-5 | マイルストーン合意 | 開発・テスト・移行のスケジュール | リリース日程 |

## 3. ヒアリングスケジュール案

```
Week 1: Phase A（事業要件）
  - Day 1-2: 事前資料配布 + レビュー依頼
  - Day 3-4: ヒアリング実施（1-2 時間）
  - Day 5:   議事録共有 + 不明点フォロー

Week 2: Phase B（技術要件）
  - Day 1-2: 事前資料配布（認可設計 + JWT クレーム案）
  - Day 3-4: 各システムチームとの個別ヒアリング（1 時間 × N チーム）
  - Day 5:   要件統合 + 矛盾点の洗い出し

Week 3: Phase C（運用・セキュリティ要件）
  - Day 1-2: 事前資料配布（DR 設計 + セキュリティ設計案）
  - Day 3-4: インフラ・セキュリティチームとのヒアリング（2 時間）
  - Day 5:   要件ドラフト作成

Week 4: Phase D（最終判断）
  - Day 1-3: 要件定義書ドラフトのレビュー
  - Day 4:   判断会議（Cognito / Keycloak 選定 + ロードマップ合意）
  - Day 5:   要件定義書の確定・承認
```

## 4. ヒアリング時の注意事項

### 4.1 よくある落とし穴

| 落とし穴 | 対策 |
|---------|------|
| 「とりあえず全部対応して」 | 優先度を付けて段階的に対応する方針を提示 |
| 「Cognito でいいんじゃない？」（安易な判断） | MAU 規模・IdP 種類・カスタマイズ要件で判断材料を提示 |
| 「Keycloak は運用が大変」（先入観） | PoC での運用実績（障害注入・復旧）を提示 |
| 要件が発散する | PoC で検証済みの範囲を基準線として提示 |
| 「セキュリティは後で考える」 | 認証基盤のセキュリティは後付けが困難であることを説明 |

### 4.2 PoC 結果の効果的な提示方法

1. **デモを見せる**: 実際のログインフロー（Cognito / Keycloak 両方）
2. **コスト表を見せる**: MAU ごとの月額コスト比較グラフ
3. **障害テスト結果を見せる**: ECS 停止 → 復旧のタイムライン
4. **認可フローを見せる**: JWT の中身を実際にデコードして見せる
