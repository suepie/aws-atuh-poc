## Excel へのインポート手順

### 方法 1: コピペ（推奨、最速）

1. TSV ファイルをエディタ（VS Code / メモ帳 / TextEdit）で開く
2. 全選択（Cmd+A / Ctrl+A）→ コピー（Cmd+C / Ctrl+C）
3. Excel の空白シートで A1 セルを選択
4. ペースト（Cmd+V / Ctrl+V）
5. 列ごとに自動分割される

### 方法 2: ファイルから直接開く

1. Excel メニュー → 「ファイル」→ 「開く」
2. ファイルの種類を「すべてのファイル」に変更
3. `.tsv` ファイルを選択
4. テキストインポートウィザード → タブ区切り → 完了

### 方法 3: データタブからインポート（最も柔軟）

1. Excel メニュー → 「データ」→ 「テキストから」
2. ファイルを選択 → タブ区切り指定
3. 列の型を個別指定可能（例: 「§」列を数値として）

## ヒアリング回（M1 / M2 / M3）— 3 回会議への分割計画

> **要件定義会議が 3 回**ある前提で、127 項目を **依存関係と負荷バランス**を考慮して分割した計画。Excel フィルタで `ヒアリング回 = M1` のように絞り込み、各会議用の質問リストを抽出できる。

### 第 1 回（M1）— 前提合意 + プラットフォーム判定軸（32 件、🔥 22 件）

| サブ§ | 件数 | 主な内容 |
|---|:-:|---|
| §1.1〜§1.5 | 6 | 前提合意（利用者カテゴリ / 採用シナリオ / Broker / ブランディング 2 軸）|
| §2.1 規模・成長 | 7 | **A-15 顧客企業数（プラットフォーム決定）** / MAU / システム数 |
| §2.2 規制・コンプラ | 5 | **C-201 FIPS（RHBK 決定）** / SOC2 / GDPR |
| §2.7 移行・リリース | 4 | 既存認証 / リリース時期 / マイルストーン |
| §3.1 マスター表 B | 7 | **B-200 IdP 統合表**（LDAP 直結 → Keycloak 確定）|
| §3.2 マスター表 C | 1 | **B-100 アプリ表**（K1〜K8 → Keycloak 確定）|
| §3.3 既存環境移行 | 2 | 既存 AWS / パスワードハッシュ |

**狙い**: プラットフォーム選定の **8 割を決着**。後続会議の前提を確定。
**対象者**: PO / 事業企画 + テックリード + 情シス
**時間目安**: 2 時間

### 第 2 回（M2）— 機能要件詳細（44 件、🔥 2 件）

| サブ§ | 件数 | 主な内容 |
|---|:-:|---|
| §2.3 マルチテナント | 10 | テナント分離粒度 / Organization / 複数所属 |
| §2.4 ユーザー管理 | 11 | SCIM 採否 / セルフサービス / アカウントリンク |
| §2.5 SSO・ログアウト方針 | 9 | SSO 範囲 / ログアウトレイヤー / 信頼レベル |
| §2.6 MFA 適用ポリシー | 4 | 必須範囲 / 外部 IdP 信頼 / AAL レベル |
| §3.5 顧客 IdP 属性マッピング | 7 | 突合せキー / 属性更新 / 退職反映 SLA |
| §4.2 認可・JWT | 3 | 必須クレーム / 認可粒度 / ロール体系 |

**狙い**: 機能要件中核を確定。プラットフォーム前提が決まっているため議論が空転しない。
**対象者**: 開発チーム / テックリード中心
**時間目安**: 2 時間

| サブ§ | 件数 | 主な内容 |

**狙い**: 技術詳細 + NFR + **最終意思決定**。多くが具体値で進めやすい。
**対象者**: インフラ / SRE / セキュリティ + 意思決定者
**時間目安**: 2 時間

### 順序設計の根拠

```
第 1 回 ──┬─→ プラットフォーム選定方向性 ──┬─→ 第 2 回
         ├─→ 規模軸（A-15）             │
         ├─→ コンプラ範囲                │
         └─→ スケジュール                ┴─→ 第 3 回
                                          ↑
第 2 回 ─→ ポリシー方針 + JWT 設計 ────────┘
```

- 第 1 回が決まらないと第 2-3 回が空転（プラットフォーム未定での詳細議論は無駄）
- 第 2 回のポリシー方針が決まらないと第 3 回の技術仕様が決まらない

### Excel での活用

**フィルタ操作**：
1. オートフィルタを設定
2. **E 列「ヒアリング回」**で絞り込み → 各会議用質問リストが抽出できる
3. 例: `M1` で絞り込むと第 1 回で聞くべき 32 件のみ表示

**ピボットテーブル**：
- 行: サブ§
- 列: ヒアリング回（M1/M2/M3）
- 値: ID のカウント
- → 各会議の質問配分が可視化される

**ステータス管理**：
- 第 1 回終了時点で M1 件すべての状態が ✅ になっていることを確認
- 残った ⏳ や ⚠ は次回持ち越し / 別途確認

## 推奨する Excel 設定

### オートフィルタを設定
1. ヘッダ行を選択 → 「データ」→ 「フィルタ」
2. § / Phase / 優先度 / 状態 列で絞り込み可能に

### 条件付き書式で優先度を色分け
- 🔥: 赤背景 / 白文字
- 🟡: 黄背景
- 🟢: 緑背景

### 状態列にプルダウン設定
1. J 列を選択
2. 「データ」→ 「データの入力規則」→ リスト
3. 元の値: `⏳,✅,⚠,❌`

### 詳細参照をハイパーリンク化
H 列の値を `=HYPERLINK("ファイルパス", "ファイル名")` 形式に変換すると、クリックで対応する md ファイルが開く（要 Excel の Hyperlink 機能対応）。

### ピボットテーブルで集計
1. 「挿入」→ 「ピボットテーブル」
2. 行: §（または Phase）
3. 列: 状態
4. 値: ID のカウント
5. → 進捗ダッシュボードができる

## マスター表 B / C の使い方

### マスター表 B: 事業者・顧客 IdP 統合表（1 + N 行）

**事業者行 1 行 + 顧客行 N 行**を 1 シートに統合。区分列で事業者 / 顧客を区別、用途列で対象ユーザー（P-1 / P-3）を明示。

| 列 | 内容 |
|---|---|
| **区分** | **事業者** / 顧客 |
| **企業名** | 弊社（事業者行）/ Acme Corp 等（顧客行）|
| **用途** | P-1 基盤運用管理者の認証（事業者行）/ P-3 一般従業員の認証（顧客行）|
| **IdP 製品（列 X）** | A-1〜A-3 / B / C-1〜C-2 / D / E / F / G / H / I / J / K / L / M / N / O |

> **記入ヒント**: 事業者行（行 0）では列 V（テナント分離）と列 Z（接続経由）は「-」（弊社は顧客テナントではない）。**dogfood ケース**（弊社が自社向けにも本基盤を利用）は、弊社を行 0（事業者、P-1 用）+ 別行（顧客、P-3 用）の 2 行で記入。

顧客企業ごとに 1 行ずつ追加。

| 列 | コード | 選択肢 |
|---|---|---|
| **列 X**: IdP 製品 | A-1〜A-3 / B / C-1〜C-2 / D / E / F / G / H / I / J / K / L / M / N / O | Entra/Okta/Auth0/Google/HENNGE/AD/IdP なし 等 |
| **列 Y**: 接続プロトコル | α / β / γ / δ | OIDC / SAML / LDAP / 独自 |
| **列 Z**: 接続経由 | ① / ② / ③ / ④ | 直結 / AD FS / Entra Connect / AD 直結 |
| **列 W**: SCIM 対応 | ✅ / ⚠ / ❌ / ❓ | 標準対応 / 上位ライセンス / 未対応 / 不明 |
| **列 V**: テナント分離 | L1 / L2 / L3 | 完全集約 / 論理分離 / 物理分離 |

詳細選択肢: [hearing-script/02-idp-federation.md マスター表 B](hearing-script/02-idp-federation.md)

#### 自動判定（表 B から）

| 表 B の状態 | 帰結 |
|---|---|
| 列 Y に **γ LDAP 直結** が 1 件でも | **Keycloak 必須化** |
| 列 V に **L3 物理分離** が 1 件でも | 追加コスト発生（B-607 連動）|
| 列 W に **❌ 未対応** が多数 | B-401 SCIM 採否で「顧客選択制」推奨 |

### マスター表 C: アプリ・システム構成（M 行 = アプリ数）

御社のアプリ・システムごとに 1 行ずつ追加。**Cognito vs Keycloak 選定の決定打**。

| 列 | コード | 選択肢 |
|---|---|---|
| **列 P**: クライアント種別 | a / b / c / d / e / f / g | SPA / SSR / モバイル / M2M / CLI・IoT / Backend API / SAML SP |
| **列 Q**: SPA 認証方式 | α / β / γ / — | BFF / PKCE 直接 / 段階移行 / 該当なし |
| **列 R**: Backend API 経路 | ① / ② / ③ / ④ / ⑤ | API GW+Lambda Authorizer / ALB+ECS / Service Mesh / サーバー内完結 / Cognito Authorizer |
| **列 S**: 特殊要件フラグ ☑ | K1 / K2 / K3 / K4 / K5 / K6 / K7 / K8 | Token Exchange / Device Code / mTLS / DPoP / SAML 発行 / UMA / Back-Channel Logout / Access Token Revocation |
| **列 T**: 既存ローカル認証 | N / M1 / M2 / M3 / M4 | なし / 段階移行 / 並行稼働 / 即時切替 / 維持 |

詳細選択肢 + 補足 1〜5（BFF/JWT 検証/Knockout 条件 K1〜K8 技術根拠/業界トレンド/FAQ）: [hearing-script/01-auth-flow.md](hearing-script/01-auth-flow.md)

#### 自動判定（表 C から）

| 表 C の状態 | 帰結 |
|---|---|
| 列 S に **K1〜K8 のいずれか 1 件**でも☑ | **Keycloak 必須化確定** |
| 列 P に **e CLI/IoT** が含まれる | 列 S K2 暗黙☑ |
| 列 P に **g SAML SP のみ** が含まれる | 列 S K5 暗黙☑ |
| 列 T に **M2 / M3 / M4** が含まれる | 移行戦略策定が必要 |

## ヒアリング推奨順序（Excel フィルタの順番）

1. **§ = 1** で絞り込み: 前提合意 6 項目すべて確認（後続全質問の範囲規定）
2. **§ = 3** で絞り込み: 既存接続元・3 大マスター表を埋める（Cognito vs Keycloak 事実上確定）
3. **§ = 2、優先度 = 🔥** で絞り込み: サービス要件の最優先項目
4. **§ = 5、優先度 = 🔥** で絞り込み: NFR の最優先項目
5. **§ = 4** で絞り込み: 技術仕様の詳細
6. 残り（🟡 / 🟢）を順次

## メンテナンスについて

- [hearing-checklist.md](hearing-checklist.md) が **SSOT**（Single Source of Truth）
- TSV ファイルは hearing-checklist.md からのスナップショット（再生成可能）
- 質問の追加・削除があった場合、TSV を再生成
- Excel 上で記入した「回答」「状態」列の保持は、TSV を上書きせず Excel ファイル側で管理推奨
