## C-3: 運用体制

---

### 【サポート体制】 (C-301, 🔥)

本基盤のサポート体制についてご希望をご教示ください:
- 24/7 サポート必須（金融 / 規制業種で典型）
- 営業時間のみ
- 不要（自社運用で完結）

**目的**: **24/7 必須の場合、Cognito（AWS Enterprise Support）または Keycloak の RHBK（Red Hat 24/7 サポート）が選択肢**となります。OSS Keycloak の場合は自社運用となり、SLA 保証が困難となります。

---

### 【既存の Red Hat 利用実績】 (C-302, 🔥)

既存の Red Hat 製品利用実績についてご教示ください:
- OpenShift 利用中（規模 / 用途）
- RHEL 利用中
- 既存契約なし

**目的**: **既存契約がある場合、RHBK 採用の追い風**となります（既存 Red Hat 契約への追加、運用ノウハウの活用、サポート窓口の一元化）。新規契約の場合は調達プロセスから検討が必要です。

---

### 【RHBK サブスクリプション予算】 (C-303, 🔥)

RHBK サブスクリプションに対する予算枠についてご教示ください。
年間予算の目安: $15K〜$90K 規模が一般的（コア数 × ノード数で変動）。
Yes / No、もしくは予算上限値でお答えいただけますと幸いです。
**目的**: RHBK 採用判断の予算的実現性確認。Keycloak OSS（無償）との TCO 比較、長期コストへの影響評価に必要な情報です。

---

### 【監視ツール】 (C-304, 🟡)

既存の監視・ログ管理ツールをご教示ください:
- CloudWatch（AWS 標準）
- Datadog
- Grafana / Prometheus
- Splunk
- New Relic
- その他

**目的**: 本基盤との統合方針、メトリクス / ログ送信先の確定、Keycloak の Micrometer / Prometheus 統合、Cognito CloudWatch Logs Insights の活用判断に必要な情報です。

---

### 【バージョンアップ方針】 (C-305, 🟡)

Keycloak / RHBK のバージョンアップ方針をご教示ください:
- LTS のみ採用（RHBK の長期サポート版）
- 最新追従（Keycloak OSS 月次リリース）
- 任意（プロジェクト都合で判断）

**目的**: バージョンライフサイクル管理、メンテナンス窓設計、Keycloak 26.5.4 リグレッション（[Issue #46605](https://github.com/keycloak/keycloak/issues/46605)）等の既知問題回避戦略に必要な情報です。

---

### 【変更管理プロセス（クロスアカウント前提）】 (C-306, 🟡)

**前提**: 本基盤は **共通基盤専用 AWS アカウント** に配置されます（御社の各アプリの AWS アカウントとは別）。そのため、共通基盤の構成変更には御社アプリチームから共通基盤運用チームへの **申請・調整プロセス** が発生します。

本基盤への変更タイプ別の承認フローをご教示ください:

**変更タイプ別の頻度想定**（[§NFR-6.4 構成変更プロセス](../proposal/nfr/06-operations.md#nfr-64-構成変更プロセスクロスアカウント) より）:
- **App Client 追加・削除**（新規アプリ立ち上げ時、月 1-2 件想定）
- **IdP 追加・削除**（新規顧客オンボーディング時、月 N 件想定）
- **クレーム / 属性スキーマ追加**（アプリ認可要件追加時、月 1 件未満）
- **Realm / User Pool 設定変更**（PW ポリシー / セッション TTL 等、半期 1 件）
- **緊急対応**（侵害対応 / 障害時、不定期）

確認事項:
- **申請窓口**: Git PR ベース推奨（本基盤の標準） / メール / チケットシステム
- **承認者**: 基盤運用 / 情シス / セキュリティ / 御社側担当
- **承認 SLA**: 変更タイプ別（PR Merge まで 1 営業日 / 2-3 営業日 / 1-2 週間 等）
- **緊急対応プロセス**（インシデント時の Fast Track）: on-call 単独承認 / staging skip 可否
- **アプリチームからの直接 PR 提出を許可するか**: はい（外部 contributor）/ いいえ（基盤運用が代理提出）

**目的**: 顧客追加リードタイム < 1 営業日の達成、Terraform IaC の運用フロー、CI/CD パイプライン設計、変更履歴の監査要件への適合性確認、緊急対応の Fast Track ルート確定に必要な情報です。**クロスアカウント前提のため、申請プロセスの軽量化（Git PR ベース）が運用負荷を大きく左右します**。

## Phase D: 最終判断会議

---

### 【プラットフォーム選定】 (D-1, 🔥)

最終的なプラットフォーム選定について、判断をお願いいたします:
- **Cognito Lite**: 基本機能のみ、無料
- **Cognito Essentials**: WebAuthn / Managed Login / パスワード履歴対応
- **Cognito Plus**: + 脅威保護 / 適応認証 / 漏洩パスワード検出
- **Keycloak OSS**: 自社運用、無償、フル機能
- **Red Hat build of Keycloak（RHBK）**: 商用サポート、FIPS 140-2 対応

[ADR-017](../../adr/017-platform-selection.md) / [ADR-018](../../adr/018-rhbk-decision.md) を参照してご決定いただけますと幸いです。
**目的**: 本ヒアリングの最終アウトプット。これまでの回答内容に基づき、本基盤の実装プラットフォームを確定するための判断です。

---

### 【段階的移行 vs ビッグバン】 (D-2, 🔥)

既存認証システムからの移行戦略についてご判断ください:
- **段階移行**: 顧客 / システム単位で段階的に移行（リスク低、期間長）
- **並行稼働**: 一定期間、新旧両基盤を並行運用
- **ビッグバン**: 一斉切替（リスク高、期間短）
- **新規のみ**: 既存はそのまま、新規顧客 / システムのみ新基盤

**目的**: 移行プロジェクトの全体計画策定、データ移行ツールの設計、運用切替リハーサルの必要性判断に必要な情報です。

---

### 【運用体制の確定】 (D-3, 🔥)

本基盤の運用体制についてご判断ください:
- **専任**: 専門チームを編成（人数 + スキルセット）
- **兼任**: 既存インフラチームが兼任
- **外部委託**: MSP / SI ベンダーに委託

**目的**: SLA 達成可能性の確認、24/7 体制への適合性、コスト見積（人件費 / 委託費）、Keycloak OSS（自社運用前提）vs RHBK（Red Hat サポート活用）の判断に必要な情報です。

---

### 【予算の確定】 (D-4, 🔥)

年間運用予算の枠をご判断ください。
- 初期構築費用（CAPEX）
- 年間運用費用（OPEX）= AWS 利用料 + Cognito MAU 課金 / RHBK サブスク + 人件費

具体的な予算枠でお答えいただけますと幸いです。
**目的**: TCO 試算との突合せ、損益分岐点（Cognito vs Keycloak: 約 17.5 万 MAU、Plus ティア時 約 7.5 万 MAU）への適合性判断、プラットフォーム最終決定の経済的根拠に必要な情報です。

---

### 【リリーススケジュール】 (D-5, 🔥)

設計 → 開発 → テスト → 移行の各マイルストーンの目標日をご判断ください:
- 設計完了
- 開発完了
- 受入テスト完了
- 移行開始
- 全顧客移行完了
- 旧基盤廃止

**目的**: プロジェクト全体スケジュールの確定、各フェーズの工数見積、必要なリソース（人員 / インフラ）の手配に必要な情報です。

---

### 【Identity Broker パターン採用前提合意】 (D-6, 🔥)

本基盤のアーキテクチャ前提として、**Hub-and-Spoke 型 Identity Broker パターン**を採用することにご合意いただけますか。
- 合意（業界標準パターン、本基盤の前提として確定）
- 異論あり（別アーキテクチャを検討したい）
- 別案検討（具体的な代替案の提示が必要）

業界標準パターンであり、本基盤の[§C-1 アーキテクチャ](../proposal/common/01-architecture.md)で詳述しています。
**目的**: 本基盤の根本アーキテクチャの最終合意。Broker パターン採用は[§FR-2](../proposal/fr/02-federation.md) の要件から構造的に導出される必然ですが、明示的な合意プロセスとして確認します。**「異論 / 別案検討」の場合は、本基盤の存在意義そのものの再検討が必要**となります。
