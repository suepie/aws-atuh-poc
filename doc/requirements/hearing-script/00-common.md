### 【受け入れる利用者カテゴリ】 (A-5-2, 🔥)

> **問いの位置づけ**: 本基盤が認証を引き受ける「ユーザー集合」（P-1〜P-6 のうちどれを含めるか）を確定する **後続全質問の範囲を規定する最上位判断**。フェデ運用のみか、ローカル受け入れも行うかの根本選択。
> **回答で決まること**: ①ローカルユーザー管理 UI の必要性（P-4 含む場合は必須）/ ②本基盤の MAU 課金規模 / ③Plus ティア侵害検出の対象スコープ / ④Break Glass 設計の規模 / ⑤A-5-3 採用シナリオの選択範囲 / ⑥A-6 / B-200 / C-204 系質問の深掘り要否

#### なぜこれを今聞くのか

本基盤は「**誰の認証を引き受けるか**」次第で、ローカルユーザー管理 UI / MFA 登録フロー / パスワードリセット / Break Glass フローの設計が**根本的に変わります**。**特に P-4（IdP を持たない顧客従業員）を含めるかは、ローカルユーザー管理機能の有無を左右する最重要判断**で、後から追加すると認証 DB スキーマ・MFA 登録 UI・パスワードリセット API すべて新設になります。事前に確定することで、無駄な実装を避けつつ十分なカバレッジを担保できます。

#### 比較イメージ（含めるカテゴリと運用負荷）

| カテゴリ | 該当例 | 含めた場合の運用負荷 |
|---|---|---|
| **P-1** 基盤運用管理者（弊社運用 / 御社代行） | 弊社運用組織の認証 | 必須（ほぼ全シナリオ）|
| **P-2** 顧客企業のテナント管理者 | 顧客企業の管理者ユーザー | 中（A-11 系で詳細）|
| **P-3** 顧客一般従業員（**フェデ**）| 顧客 IdP 経由のユーザー | **大**（MAU 課金最大、B-200 で詳細）|
| **P-4** IdP なし顧客従業員（**ローカル**）| 顧客が IdP を持たない場合 | **最大**（C-204 系のローカル認証機能すべて該当）|
| **P-5** Break Glass 用ローカル管理者 | 緊急時 / IdP 障害時の最後の砦 | 小（数名）|
| **P-6** B2C コンシューマー | 一般消費者向け SaaS | 大（該当業態のみ、Plus ティア対象）|

#### 質問

本基盤を経由して認証を受けるユーザーのカテゴリをご教示ください。本基盤の設計では利用者を 6 つのカテゴリに分類しています（[§FR-1.2.0.0](../proposal/fr/01-auth.md) 参照）。複数該当する場合は全てお選びいただけますと幸いです:

- **P-1: 基盤運用管理者**（御社運用チーム / 弊社が運用代行）→ Break Glass 用ローカル + 御社内 IdP 経由
- **P-2: 顧客企業のテナント管理者**（自社テナント運用、A-11 系で詳細）
- **P-3: 顧客企業の一般従業員**（**顧客 IdP 経由のフェデユーザー**、A-6 / B-200 で詳細）
- **P-4: IdP を持たない顧客のユーザー**（**本基盤内のローカルユーザー**、C-204 で詳細）
- **P-5: Break Glass 用ローカル管理者**（緊急時 / IdP 障害時の最後の砦、数名のみ）
- **P-6: B2C コンシューマーユーザー**（一般消費者向けサービス、該当しない場合が多い）

**目的**: 本基盤が受け入れるユーザーの全体像確定。**P-3 のみなら純粋なフェデ運用**、P-4 が含まれるなら**ローカルユーザー管理機能（パスワードポリシー / リセット / MFA 登録 UI 等）が必須**、P-1 / P-5 の Break Glass 設計の規模感も決定。後続の A-6（顧客 IdP 分布） / B-200 系（IdP 詳細） / C-204（パスワードポリシー）のスコープを規定する最上位質問です。本回答により、本基盤の構成・運用負荷・コスト構造が大きく変わります。

### 【ローカルユーザー受け入れ範囲（採用シナリオ）】 (A-5-3, 🔥)

> **問いの位置づけ**: A-5-2 で確定したカテゴリのうち、**「ローカルユーザーとして本基盤が直接受け入れる範囲」**を 4 つのシナリオ（α/β/γ/δ）から選ぶ。事業戦略・MAU 課金・運用負荷の根本パラメータ。
> **回答で決まること**: ①ローカルユーザー総数（MAU 課金・侵害検出スコープに直結）/ ②顧客契約に「IdP 提供必須」条件を入れるか / ③ローカルユーザー管理機能の規模（パスワードリセット UI / MFA 登録 UI / Self-Service Account Console）/ ④Plus ティア（侵害検出）の費用対効果。

#### なぜこれを今聞くのか

本基盤の **事業戦略の根本パラメータ**です。**「IdP を持たない顧客を受け入れるか」**＝**「IdP 導入を顧客契約条件にできるか」**で、事業戦略・顧客の獲得幅・運用負荷・コスト構造のすべてが変わります。

- 業界推奨は **γ シナリオ**（管理者層のみローカル、エンドユーザーは IdP 強制）— エンタープライズ B2B SaaS の主流
- α シナリオは B2C を含む大規模展開で必要
- δ シナリオは規制業種専用 SaaS で稀

後から「IdP なし顧客も受けるようにしたい」となると、ローカルユーザー管理機能 + MAU 課金影響を含めて大規模な変更になります。

#### 比較イメージ（4 シナリオの差分）

| シナリオ | ローカル受入カテゴリ | ローカルユーザー規模 | MAU 課金影響 | IdP 必須化 | 採用シーン |
|---|---|---|---|---|---|
| **α 全カテゴリ** | P-1〜P-6 全て | **最大** | 最大 | 不要 | B2C を含む大規模 SaaS |
| **β 管理者 + IdP なし顧客** | P-1/P-2/P-4/P-5 | 中 | 中 | 部分 | IT 成熟度がまちまちな B2B |
| **γ 管理者層のみ**（**業界推奨**）| P-1/P-2/P-5 | 小 | 最小（フェデは別課金）| **必須化（IdP 提供義務）** | エンタープライズ B2B SaaS |
| **δ Break Glass のみ** | P-1/P-5 | 最小（数名）| 最小 | **完全強制** | 規制業種専用 SaaS |

#### 質問

本基盤が受け入れる**ローカルユーザー**（顧客 IdP 経由のフェデではなく、本基盤内に直接登録されるユーザー）の範囲をご教示ください。
本基盤の設計は以下 4 つのシナリオから選択する構造になっています（[§FR-1.2.0.0](../proposal/fr/01-auth.md) 参照）:

- **α 全カテゴリ受け入れ**: P-1〜P-6 全て本基盤がローカルユーザーとして受け入れる
  - ローカルユーザー規模: **最大**（顧客従業員総数）
  - 運用負荷・MAU 課金とも最大、Plus ティアの侵害検出対象も最多
  - 採用シーン: B2C を含む大規模 SaaS

- **β 管理者 + IdP なし顧客**: P-1 / P-2 / P-4 / P-5 を受け入れ、P-3 はフェデのみ
  - ローカルユーザー規模: **中**（IdP なし顧客従業員 + 管理者層）
  - 採用シーン: 顧客の IT 成熟度がまちまちな B2B SaaS

- **γ 管理者層のみ**（**業界推奨**）: P-1 / P-2 / P-5 のみローカル、P-3 / P-4 はフェデ強制
  - ローカルユーザー規模: **小**（顧客数 × 数名 + 弊社 Break Glass 数十名）
  - **IdP を持たない顧客には IdP 導入を契約条件として要求する**
  - 運用負荷・MAU 課金最小、フェデユーザーは MAU 課金されるが侵害検出対象が圧縮
  - 採用シーン: エンタープライズ B2B SaaS

- **δ Break Glass のみ**: P-1 / P-5 のみ（数名）、全顧客ユーザーは IdP 経由フェデ強制
  - ローカルユーザー規模: **最小**（数名）
  - 顧客に対して「IdP 必須」の事業契約が前提
  - 採用シーン: 規制業種専用 SaaS、最高セキュリティ

**目的**: 本基盤の**事業戦略・運用負荷・コスト試算の根本**。シナリオ次第で:
- ローカルユーザー管理 UI の必要性（パスワードリセット / MFA 登録 UI 等）
- MAU 課金規模（ローカル vs フェデで単価が同じだがティアの選択に影響）
- Plus ティア（侵害検出）の対象スコープ（α で最大、γ/δ で最小）
- 顧客契約での「IdP 提供必須」条件の要否（γ/δ で前提）

が決まります。**業界推奨は γ**（管理者層のみローカル、エンドユーザーは IdP 強制）です。

### 【インフラ運用者カテゴリ確認】 (A-5-4, 🔥)

> **問いの位置づけ**: A-5-2/A-5-3 は **本基盤の認証経由のユーザー** の確認だが、本問は **本基盤のインフラ自体を操作する別系統の人**（AWS IAM 等で認証、本基盤の Cognito/Keycloak 経由ではない）を確認する。
> **回答で決まること**: ①各カテゴリの認証経路設計（AWS IAM / kubectl / Datadog SSO 等）/ ②MFA 要件（I-1〜I-4 必須） / ③最小権限ポリシー設計 / ④Bastion / Privileged Access Management の必要性 / ⑤緊急対応プロセス / ⑥C-301（サポート体制）との整合性。

#### なぜこれを今聞くのか

本基盤の**運用設計の網羅性確保**のために、**「誰が本基盤を operate するか」**を事前確定が必要です。**「I-X の運用者が不在」の場合、外部ベンダー丸投げ or 自社運用なし**を意味するため、サポート契約（C-301）と整合させる必要があります。後から「監視 SRE は別チーム」「セキュリティ監査ログ要件あり」となると、**ログ要件・SIEM 連携・IAM 設計**を後付けで増設することになります。

なお本問の**本基盤の Cognito / Keycloak 設計には直接影響しません**（別系統の認証経路）。しかし**運用設計と監査ログ要件で必須の整理**になります。

#### 比較イメージ（5 カテゴリの認証経路と権限）

| カテゴリ | 操作対象 | 認証経路 | 推奨権限 | 想定不在時 |
|---|---|---|---|---|
| **I-1** AWS インフラ運用者 | Cognito / Lambda / VPC / API GW / DynamoDB | AWS IAM（IAM Identity Center）、**MFA 必須** | 最小権限（環境別 Role） | 外部 SI 委託の AWS 構築運用 |
| **I-2** Keycloak / RHBK 運用者 | EKS / ECS / OpenShift 上の Keycloak | AWS IAM + kubectl auth / OpenShift OIDC | Cluster Admin → 環境別 RBAC | Cognito 採用時は不要 |
| **I-3** 監視・SRE 担当 | CloudWatch / Datadog / Grafana / Splunk | AWS IAM / Datadog SSO、**Read-Only 推奨** | アラート受信 / ダッシュボード閲覧 | 外部 SOC / MSP 委託 |
| **I-4** セキュリティ監査者 | CloudTrail / Cognito 監査ログ / SIEM | AWS IAM / SIEM 認証、**Read-Only 強制** | 監査ログ閲覧のみ | 監査要件未確立 |
| **I-5** ベンダー / SI サポート | Red Hat Support / AWS Support / 外部 SI | サポートチケット経由 / **IAM Role STS 一時付与**（24-72h） | 期間限定の必要最小 | 自社運用のみ（外部 SI なし）|

### 用語の前提 — ブランディングカスタマイズの 2 軸（事前にすり合わせ）

ブランディングカスタマイズは **2 つの独立した軸** で考えると整理しやすくなります。両者は理論的には独立しており、それぞれ Yes/No で答えていただけば組合せでパターンが自動的に決まります。

| 軸 | 何の話か | 例 | 担当 |
|---|---|---|---|
| **軸 1: アプリ別カスタマイズ**（A-11）| アプリごとに認証基盤側のログイン画面を変えるか? | 経費精算 vs 決済管理 | 認証基盤側（`client_id` ベース）|
| **軸 2: 顧客別カスタマイズ**（A-11-α）| 顧客企業ごとに認証基盤側のログイン画面を変えるか? | Acme vs Globex | 認証基盤側（`tenant_id` ベース）|

#### 2 軸の組合せからパターンを自動判定

| 軸 1（アプリ別） | 軸 2（顧客別） | 結果パターン | 採用シーン |
|:---:|:---:|:---:|---|
| ❌ No | ❌ No | **パターン A** | アプリ画面のみカスタマイズ（最シンプル、Slack/Notion 型）|
| ✅ Yes | ❌ No | **パターン A'** | 経費精算 vs 決済等でアプリ間差別化（Auth0/Entra/Okta 型、業界主流）|
| ❌ No | ✅ Yes 部分 | **パターン B** | 顧客別ブランディング（Cognito 20 上限、規制業種）|
| ✅ Yes | ✅ Yes | A' × B 複合 | アプリ × 顧客の組合せ（Cognito ほぼ不可、Keycloak 必須）|
| - | ✅ Yes 完全分離 | **パターン C** | Pool/Realm 分離、SSO 喪失リスクあり、Enterprise プラン |

詳細根拠と公式ソース引用は [branding-strategy-evidence.md](../../common/branding-strategy-evidence.md) を参照ください。

**【ご参考】認証前後で利用可能な識別子の違い**:
- 認証前（ログイン画面）: **`client_id` パラメータ**（OAuth 標準、軸 1 で利用）— JWT はまだ発行されていません
- 認証後（アプリ画面）: **JWT クレーム `tenant_id`**（軸 1 / 軸 2 でアプリ側が利用）

### 【ブランディング軸 1: アプリ別カスタマイズ要否】 (A-11, 🔥)

> **問いの位置づけ**: 認証基盤側ログイン画面の **アプリ別カスタマイズ要否**を確定する。軸 2（A-11-α 顧客別）と独立した軸で、両者の Yes/No 組合せで実装パターン A / A' / B / C が自動判定される。
> **回答で決まること**: ①Cognito Managed Login Branding 機能の利用要否 / ②Cognito ティア選定（Essentials+ 必要かどうか）/ ③Lambda 工数（Pre Token Generation で client_id ベース差替）/ ④URL 設計（共通 1 つ vs アプリ別）/ ⑤B-612 / B-703-3 等のブランディング詳細質問の深掘り要否。

#### なぜこれを今聞くのか

**アプリ別ブランディング**は業界主流（Auth0 / Entra / Okta / Cognito Managed Login Branding）ですが、**実装方式の選択肢は 2 つ**で運用負荷が大きく違います：

- **A' 認証基盤側でカスタマイズ**: Cognito Managed Login Branding（**20 Style 上限**）or Keycloak Theme — 統一的だが上限あり
- **A 認証基盤共通 + アプリ側動的差替**: アプリの DOM で差替 — シンプル、上限なし、Slack/Notion 型

**後から「アプリ別ブランディングが必要」となると、認証基盤側の Lambda 改修 + ティア変更 + Style 設計が必要**になります。最初に方向性合意するのが圧倒的に低コストです。

#### 比較イメージ（パターン A vs A'）

| 観点 | パターン A（No、アプリ側動的差替）| パターン A'（Yes、認証基盤側カスタマイズ）|
|---|---|---|
| **認証基盤側 UI** | 共通 1 セット | アプリ別（client_id ベース）|
| **実装場所** | アプリ側 JS / CSS / SSR | Cognito Managed Login Branding / Keycloak Theme |
| **Cognito ティア** | Lite で OK | **Essentials+ 必要** |
| **上限** | なし | **Cognito 20 Style / Pool**（Keycloak は制限なし）|
| **業界例** | Slack / Notion / Microsoft 365 | Auth0 / Entra / Okta（外向きサービス）|
| **運用負荷** | 小（アプリ側完結）| 中（Style ごとに認証基盤側変更）|
| **推奨** | 内部向け B2B SaaS | 顧客向け公開サービス |

#### 質問

認証基盤側のログイン画面を **アプリごと**（経費精算 / 決済管理 / 人事 等）に別デザインにする必要はございますか。

選択肢:
- **Yes**: アプリ間で異なるロゴ・配色のログイン画面を提供したい → **パターン A'**（業界主流: Auth0 / Microsoft Entra / Okta / AWS Cognito Managed Login Branding 採用）
- **No**: 全アプリ共通の認証基盤画面で十分（ログイン後のアプリ画面は各アプリで動的差替）→ **パターン A**（Slack / Notion / Microsoft 365 型、最シンプル）

**目的**: 「アプリ間で異なるブランド体験を提供したい」場合、認証基盤側のカスタマイズ（Cognito Managed Login Branding Style / Keycloak Login Theme Override）が必要となるかの判断。
- Yes（A'）の場合: Cognito Essentials+ ティア + Managed Login Branding（**20 Style 上限**）/ Keycloak Theme（制限なし）で対応
- No（A）の場合: 認証基盤側は共通 1 セット、アプリ側で JWT 解釈による動的差替で完結

**本軸は A-11-α（軸 2 顧客別）と独立**しています。両軸の Yes/No 組合せでパターンが自動的に決まります。

---

### 【ブランディング軸 2: 顧客別カスタマイズ要否】 (A-11-α, 🔥)

> **問いの位置づけ**: 認証基盤側ログイン画面の **顧客企業別カスタマイズ要否**を確定する。軸 1（A-11 アプリ別）と独立した軸。Yes 完全分離（C）は **Identity Broker パターン崩壊リスク**を伴う最重要判断。
> **回答で決まること**: ①パターン B（20 顧客上限）vs C（SSO 喪失）vs A の選択 / ②Cognito 採用時の 20 顧客上限抵触判定 / ③User Pool / Realm 分離の要否 / ④SSO の成立可否 / ⑤Custom Domain 4 個 Hard Limit との連動。

#### なぜこれを今聞くのか

**顧客別ブランディング**は B2B SaaS で頻出要件だが、実装方式により**根本的な制約・リスク**が違います：

- **A / A' アプリ側完結**: 業界標準、Custom Domain 共通 1 つ、SSO 成立、上限なし
- **B 認証基盤側で顧客別**: Cognito Managed Login Branding（**20 顧客 Hard Limit**）、Keycloak Theme（制限なし）
- **C 完全分離（Pool/Realm per 顧客）**: **SSO が成立しない**、Custom Domain 4 個 Hard Limit、Identity Broker パターンの利点喪失

**特に C パターンは設計後に変更不能** — Pool 分離は CSV 移行ですら個別ユーザー識別が変わるため再構築相当のコスト。**最初に B or C の必要性を確定が必須**。

#### 比較イメージ（パターン A / A' vs B vs C）

| 観点 | A / A'（No、アプリ側完結）| B（Yes 部分、認証基盤側）| C（Yes 完全分離）|
|---|---|---|---|
| **認証基盤側分離** | 単一 Pool / Realm | 単一 Pool / Realm + Style 分離 | **Pool / Realm per 顧客** |
| **SSO** | ✅ 成立 | ✅ 成立 | **❌ 喪失**（テナント間で別 IdP）|
| **Identity Broker パターン** | ✅ 標準 | ✅ 標準 | **❌ 崩壊**（顧客ごとに別 Broker）|
| **Cognito 上限** | なし | **20 顧客 / Pool**（Managed Login Branding Style）| **無関係**（Pool ごとに 4 Custom Domain Hard Limit）|
| **Custom Domain** | 共通 1 つ | 共通 1 つ | **顧客別**（4 個 Hard Limit）|
| **Callback URL** | 共通 + クエリ動的差替 | 共通 + クエリ動的差替 | **顧客別 URL**（allowlist 肥大化）|
| **Cognito 採用可** | ✅ 任意ティア | ✅ Essentials+（20 顧客以内）| ⚠ Enterprise プラン相当（事実上 Keycloak 必須）|
| **後戻りコスト** | 低 | 中（Cognito 21 顧客目で Keycloak 移行）| **極大**（Pool 分離は再構築相当）|

#### 質問

認証基盤側のログイン画面を **顧客企業ごと**（Acme / Globex 等）に別デザインにする必要はございますか。

選択肢:
- **No**: 顧客別ブランディングは認証基盤側では不要、アプリ側で `tenant_id` クレーム解釈で動的差替で完結 → **パターン A または A'**（業界標準、推奨）
- **Yes 部分カスタマイズ**: 顧客別ロゴ・配色を認証基盤画面でも表示したい（Cognito では **20 顧客上限**、Keycloak ならカスタム Theme で対応）→ **パターン B**
- **Yes 完全分離**: 顧客ごとに完全専用デザイン + 認証基盤を物理分離 → **パターン C**（Pool / Realm 分離、**SSO 喪失 + Identity Broker パターン崩壊リスク**、Enterprise プラン相当）

**目的**: 顧客企業ごとの認証基盤画面カスタマイズの必要性確認。
- No の場合: アプリ側で動的差替で完結、Custom Domain 共通 1 つ + Callback URL 共通で URL 肥大化なし
- Yes 部分（B）の場合: Cognito 採用時は 20 顧客上限の壁、それ以上は Keycloak 必須
- Yes 完全分離（C）の場合: **User Pool / Realm を分離するため SSO が成立しなくなる**、Identity Broker パターンの利点も失われる（[branding-strategy-evidence.md §7.D](../../common/branding-strategy-evidence.md) 参照）

**本軸は A-11（軸 1 アプリ別）と独立**しています。両軸の Yes/No 組合せでパターン A / A' / B / C が自動判定されます。

### 【アプリ側の動的差替実装責務】 (A-11-2, 🟡)

**A-11 = No または A-11-α = No**（= 顧客別差替を**アプリ側で対応**するパターン A / A'）採用時、各アプリチームが以下の実装責務を取れることをご確認ください:

- **JWT クレーム（`tenant_id`）から顧客識別** — JWT は本基盤が発行
- **ロゴ・配色・文言の動的差替** — CSS / JS / SSR いずれの方式でも可
- **`tenant_id` の改竄検証** — クエリパラメータ（`?tenant=acme`）ではなく **JWT クレームを真実の源**として使用
- **ログアウト後ランディングページの実装** — 共通 URL（例: `/post-logout`）でクエリ受け取り

選択肢:
- **責務を取れる**（パターン A / A' 採用可能）
- **一部支援が必要**（サンプルコード / SDK 提供で対応可能）
- **取れない**（A-11-α で Yes 部分（B）or 完全分離（C）への変更を再検討）

**目的**: パターン A / A' 採用の実現可能性確認（アプリ側で顧客別差替する責務を取れるか）。アプリチームの実装スキル・工数評価、本基盤側の支援範囲（サンプルコード提供 / 開発ガイド / SDK 提供等）の確定に必要な情報です。「取れない」の場合は A-11-α で B/C パターン（顧客別を認証基盤側で対応）を選択し、認証基盤側のカスタマイズコスト（Cognito 20 顧客上限 / Keycloak Theme 開発工数）を許容するかの判断が必要となります。

### 【カスタマイズレベル確認（A-11 = Yes または A-11-α = Yes 時）】 (A-11-3, 🟡)

**A-11 = Yes（アプリ別を認証基盤側で）** または **A-11-α = Yes（顧客別を認証基盤側で）** を採用される場合、認証基盤側ログイン画面に求めるカスタマイズレベルをご教示ください。

カスタマイズレベル別の対応:

**【L1-L3: 見た目・スペーシング・基本配置】** — Cognito / Keycloak 両方で対応可
- L1: ロゴ・色・フォント・影・ボーダー半径
- L2: スペーシング・要素間隔・パディング
- L3: 要素位置の選択（ロゴ位置: Header or Form 内、フォーム水平配置）※ Cognito は事前定義の選択肢内に限定

**【L4-L8: DOM 構造変更を含む高度なカスタマイズ】** — **Keycloak 必須化** または特殊対応必要
- L4: テキスト・文言変更（"Email" → "メールアドレス" 等）
- L5: 要素追加・削除（規約同意チェックボックス追加等）
- L6: 要素並び順変更
- L7: HTML 構造の完全自由書き換え
- L8: カスタム JS / 動的挙動・独自バリデーション

選択肢:
- **L1-L3 で十分**（ロゴ・配色・基本配置のみ）→ Cognito Managed Login Branding で対応可
- **L4-L8 が必要**（文言・要素追加・並び順・完全カスタム）→ **Keycloak Theme 必須**（Cognito 採用時は Custom UI 自前実装、または該当アプリ単独でアプリ側 UI に寄せる選択）
- **アプリ側カスタム UI に寄せる**（共通基盤の運用負荷回避、ただし SSO 喪失等のトレードオフあり）

**目的**: パターン A' 採用時の **プラットフォーム選定への直接影響**確認。L4-L8 が必要なら **Cognito Managed Login Branding では対応不可** のため、Keycloak への移行検討 or アプリ側カスタム UI の選択が必要となります。詳細な技術根拠・公式ソース引用・トレードオフは [branding-strategy-evidence.md §7.A カスタマイズレベル別マトリクス](../../common/branding-strategy-evidence.md) と [§7.B カスタム UI をアプリ側に寄せる選択肢](../../common/branding-strategy-evidence.md) を参照ください。
