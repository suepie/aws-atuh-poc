### 【デフォルトのログアウトレイヤー】 (B-701, 🟡)

「ログアウト」ボタンを押したとき、本基盤がどこまでセッションを破壊すべきかご教示ください:
- **L1 ローカルのみ**: 当該アプリのセッションのみ破棄
- **L2 IdP セッション破棄**: 本基盤の SSO セッションまで破棄
- **L3 フェデ連動**: 外部 IdP（Entra ID / Okta 等）のセッションも連動破棄
- **L4 Back-Channel**: 同 IdP 内の全アプリも Back-Channel で同期切断

業務シナリオ（シェア端末利用 / コンプラ要件 / 退職時遮断）からも判断できます。
**目的**: ログアウトの破壊範囲確定。**L4 が Must の場合、Cognito では実装不可能なため Keycloak 必須化**となり、プラットフォーム選定に直結します。

---

### 【フェデ連動ログアウト要否】 (B-702, 🟡)

外部 IdP（Entra ID / Auth0 等）のセッションも連動して破棄する必要はございますか。
有無でお答えいただけますと幸いです（Cognito では URL エンコード制約あり）。
**目的**: OIDC RP-Initiated Logout のフェデ連動実装、`id_token_hint` を使った IdP セッション切断の設計、Cognito 採用時の制約確認に必要な情報です。

### 【ログアウト後リダイレクト先（全体方針）】 (B-703, 🟡)

ログアウト後のリダイレクト先 URL の管理方針についてご教示ください:
- 基盤側で集中管理（全顧客の URL を運用チームが allowlist 登録）
- アプリ側で個別管理（各アプリチームが自分の URL を登録）
- 顧客側で持ち込み（顧客企業がリダイレクト先を指定）

**事前登録（allowlist）が必須**でオープンリダイレクト攻撃対策のため、任意 URL は受け付けません。
**目的**: 運用フローと責任分担の明確化、URL allowlist 管理の責任所在の確定に必要な情報です。

---

### 【許可リダイレクト URL の事前登録】 (B-703-1, 🟡)

> **※ [A-11](00-common.md) で「パターン A」を合意された場合の標準動作: 共通 URL + クエリ動的差替で 5〜10 URL/アプリで完結、Cognito 100 URL/App Client Hard Limit 内に十分収まります。**

各アプリで許可するリダイレクト URL のリストをご教示ください。
1 つのアプリで通常 5〜10 個（ログインコールバック / ログアウト完了 / エラー / セッションタイムアウト / 強制ログアウト後 等）が必要です。

- **A-11 パターン A 採用時**: 共通 URL（例: `https://app.example.com/callback`、`https://app.example.com/post-logout`）+ クエリ `?tenant=acme` で動的差替 → **顧客数に依存せず 5〜10 URL/アプリで完結**
- **A-11 B/C 採用時**: 顧客別 URL を allowlist 登録 → **顧客数 × アプリ数 × 用途数で線形拡大**（Cognito 100 URL Hard Limit に到達リスク）

**目的**: App Client / Client の事前設定および URL 設計の選定に必要な情報です。**未確定だと App Client / Client 設定不能**となります。**A-11 パターン A 合意済みなら共通 URL リストのみ確認で十分**。

---

### 【ログアウト種類別の飛び先】 (B-703-2, 🟡)

ログアウトの種類ごとに、飛び先（リダイレクト先）を変える必要はございますか:
- 通常のユーザー操作によるログアウト → アプリのログイン画面
- L4 全アプリ同期ログアウト → 「全アプリログアウト完了」案内
- 強制ログアウト（退職処理 / 管理者操作）→ 「アクセス権が削除されました」案内
- セッションタイムアウト → 「セッション切れ」案内

種類別の URL マッピングが必要か、共通の単一画面で十分か、ご希望をご教示ください。
**目的**: URL allowlist の数、アプリ側ランディングページの実装方針（種類別ページ作成 vs 単一ページのクエリ動的差替）の決定に必要な情報です。

---

### 【テナント別カスタムランディング要否】 (B-703-3, 🟢)（A-11 連動）

> **※ [A-11](00-common.md) で「パターン A」を合意された場合の標準動作: 共通 URL（`https://app.example.com/post-logout`）+ クエリパラメータ（`?tenant=acme`）でアプリ側 JS が動的に差替して対応。本質問は A-11 で B/C 選択時のみ深掘り**。

顧客企業ごとに別のログアウト完了画面（ロゴ・配色・文言の差別化）の要件をご教示ください:

- **A-11 パターン A 採用時**: アプリ側で動的差替（共通 URL）= **標準対応**、追加質問不要
- **A-11 B/C 採用時**: 顧客別 URL を allowlist 登録（URL 肥大化リスクあり、Cognito 100 URL 上限に注意）
- **なし**: 全顧客共通画面で運用

**目的**: Cognito の 100 URL Hard limit への適合性確認、アプリ側実装方針の決定に必要な情報です。**A-11 パターン A 合意済みなら、アプリ側で `tenant_id` クレーム解釈での動的差替が前提**となります。

### 【リダイレクト先での後処理要件】 (B-703-4, 🟢)

ログアウト後リダイレクト先のページで、追加の後処理が必要でしょうか:

- ローカルストレージ / Service Worker キャッシュのクリア

### 【強制ログアウト時の飛び先】 (B-703-5, 🟡)

管理者による強制ログアウトや退職処理時の飛び先について、ご希望をご教示ください。
通常のログアウトと同じ画面で良いか、それとも別の URL に「アクセス権が削除されました」等の明示メッセージを表示する設計をご希望でしょうか。
**目的**: 自発ログアウトと強制ログアウトの UX を明確に分けることで、ユーザーが混乱せず適切な行動を取れるよう設計するために必要な情報です。**自発ログアウトと別 URL にして UX 明確化を推奨**しています。

### 【Access Token Revocation 要否】 (B-704, 🟡)

> **本問は [マスター表 C](01-auth-flow.md#マスター表-c-御社アプリシステム構成リスト) 列 S K8（Access Token 即時 Revocation）に統合済**です。該当アプリで列 S K8 を☑してください。本ページの本項は**該当判定の参考情報**として残しています。

Access Token の即時無効化が必要なアプリが**1 つでも該当する場合**、表 C で該当アプリの列 S K8 を☑してください。

**判定**:
- ☑ **K8**: 規制要件、即時アクセス遮断義務、短 TTL（5-15 分）では侵害ウィンドウを許容できない
- （☑ なし）: 短 TTL で吸収可能（5-15 分のラグを許容、Refresh Token revoke で間接無効化）

**目的**: 両プラットフォームとも Access Token の個別 revoke は不可（Refresh Token revoke で間接無効化）。Cognito では `origin_jti` クレームを使った自前実装、Keycloak では Token Introspection（`/introspect`）標準提供で対応します。**「☑ K8」採用時は Cognito の自前実装コスト or Keycloak の Introspection によるパフォーマンスコストの選択**となります。

### 【管理者による強制ログアウトの粒度】 (B-705, 🟡)

基盤管理者・テナント管理者が強制的にセッションを破棄する操作について、どの粒度が必要でしょうか:
- 全ユーザー一斉ログアウト
- テナント単位（特定顧客の全ユーザー）
- 個別ユーザー

**目的**: Cognito の `AdminUserGlobalSignOut`、Keycloak の Logout API の実装スコープ、管理画面の機能設計に必要な情報です。

### 【ユーザー自身のセッション管理 UI】 (B-706, 🟢)

ユーザーが自分のアクティブセッションを確認・破棄できる UI（「他の端末をログアウト」機能）が必要でしょうか。
有無でお答えいただけますと幸いです。
**目的**: Keycloak Account Console の利用可否（標準提供）、Cognito では自前実装が必要（Lambda + DynamoDB）。開発工数の見積に直結する情報です。
