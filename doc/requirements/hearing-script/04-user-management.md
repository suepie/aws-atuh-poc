## 用語の前提（本セクションで使う JIT / SCIM / Webhook の整理）

本セクションでは以下の用語を区別して使用します（[§FR-7.4.0](../proposal/fr/07-user.md#fr-740-scim-の位置づけと本基盤のスタンス) と整合）:

| 用語 | 方向 | 契機 | 主体 |
|---|---|---|---|
| **JIT プロビジョニング** | 外部 IdP → **共通基盤側** にレコード作成（**ログインのついで**）| **ユーザーが本基盤に初回ログインした瞬間** | 基盤側（自動）|
| **SCIM プロビジョニング** | 御社（顧客） HR/IdP → **共通基盤側**（**独立 REST API**）| **御社側で人事変更（入社/異動/退職）が起きた瞬間** | 顧客 IdP（push）|
| **Webhook 通知** | **共通基盤側 → 御社の各アプリ**（**方向逆**）| **基盤側でユーザーイベントが発生した瞬間**（JIT / SCIM / 手動いずれの契機でも）| 基盤側（push）|

→ **JIT と SCIM は方向は同じ（外部 → 共通基盤）ですが、起動契機・動作タイプが真逆**（JIT = reactive・ユーザーログイン契機 / SCIM = proactive・HR 人事イベント契機の push）であり、**両方併用が標準**です。一方 **Webhook は方向自体が真逆**（共通基盤 → 御社アプリ）であり、JIT/SCIM の代替ではなく補完関係にあります。本セクションの「JIT」は基本的に **共通基盤側のユーザー DB へのレコード作成** を指します（御社 IdP 側の JIT ではありません）。

※「**JIT 管理者**」（必要時のみ管理者権限を付与、Microsoft Entra PIM 等）は別概念で、[§FR-8.3](../proposal/fr/08-admin.md) で扱います。本セクションの「JIT」は **JIT プロビジョニング** のみを指します。

### 【SCIM プロビジョニング採否】 (B-401, 🟡)

> **問いの位置づけ**: ユーザーの新規作成・属性更新・**退職時の即時無効化**を「次回ログイン契機（JIT）」で処理するか、「御社 HR/IdP 側の人事イベント契機（SCIM push）」で処理するか — 共通基盤の**プロビジョニング受信機能のスコープ**を確定する問い。
> **回答で決まること**: ①SCIM 2.0 受信 API の実装要否（Cognito 採用時は Lambda 自前実装、Keycloak はプラグイン）/ ②退職反映 SLA（JIT のみ = Access Token TTL 依存、最大 60 分 / SCIM = 即時〜数分）/ ③貴社 IdP 製品の SCIM 対応ライセンス必要性。

#### なぜこれを今聞くのか

**退職者の不正アクセス**は認証基盤の典型インシデント源です。JIT のみだと「退職処理しても再ログインがない限り基盤側レコードが残る」=「Access Token TTL 内は侵害可能」という構造的な穴ができます。SOC 2 / ISO 27001 / FFIEC 等のコンプライアンス準拠では「**退職時 N 時間以内に無効化**」が SLA として要求されることが多く、JIT だけでは担保困難です。

加えて、**この設計判断は移行後に変えると認証フロー全体の再設計**になる（基盤側 DB スキーマ・退職反映ジョブ・監査ログ要件・顧客 IdP 側ライセンスがすべて連動）ため、要件定義時点で確定が必要です。

#### 比較イメージ

| 観点 | JIT のみ | SCIM 併用 |
|---|---|---|
| **新規作成タイミング** | 初回ログイン時 | 御社 HR で入社処理した瞬間（push） |
| **退職反映** | **再ログインしない限り反映なし** | 即時〜数分（HR で削除 → push） |
| **侵害ウィンドウ**（退職後の不正アクセス）| Access Token TTL 内（15〜60 分）+ Refresh Token 期間 | ほぼゼロ |
| **御社 IdP 側必要要件** | なし | SCIM クライアント機能必須（Entra Free 不可、**Entra P1+ / Okta 全プラン / HENNGE 等**、[マスター表 B 列 W](02-idp-federation.md) で確認） |
| **本基盤の実装コスト**（Cognito 採用時）| 標準 | **Lambda + DynamoDB 自前実装**（開発工数 + 運用負担）|
| **本基盤の実装コスト**（Keycloak 採用時）| 標準 | プラグイン or 公式 SCIM SPI |
| **業界トレンド** | 中小規模 / B2C で多い | エンタープライズ B2B のデファクト |
| **規制対応** | 困難（SLA 担保不可）| 容易（即時無効化で SOC 2 / ISO 27001 等準拠）|

#### 質問

以下のどれをご希望でしょうか:
- **① JIT のみ**: ユーザーが本基盤に初回ログインした瞬間に、共通基盤側で自動レコード作成
- **② SCIM 併用**: 御社 HR / IdP で人事変更（入社・異動・退職）が起きた瞬間に、御社 IdP → 共通基盤へ自動 push 同期
- **③ 顧客選択制**: 顧客企業ごとに JIT / SCIM を選べる（顧客 IdP の SCIM 対応状況に応じて）

**併せて**: 想定退職フローの規模感（年間 / 月間 N 名）、退職反映 SLA の規制要件の有無（あれば法規名）。

### 【バルクインポート方式】 (B-403, 🟢)

新規顧客オンボーディング時、**既存ユーザーをどのように共通基盤側に投入するか**ご教示ください。
- **バルクインポート（CSV 一括投入）**: 事前に全ユーザーを基盤に取り込む。**運用開始時点で全員登録済み**
- **SCIM 経由**: 御社 IdP からの SCIM 同期で初期投入と継続同期を兼ねる
- **JIT 任せ（初回ログイン時に作成）**: 事前投入なし、ユーザーが順次ログインしてきた時点で自動作成。**移行期間中はログイン未試行ユーザーは未登録のまま**

規模感（投入予定ユーザー総数: 数百 / 数千 / 数万）も併せてお答えいただけますと幸いです。
**目的**: 初期移行設計、Cognito の AdminCreateUser API レート制限（50 RPS）への対応設計、Keycloak のインポート機能利用判断に必要な情報です。**JIT 任せの場合、未ログインユーザーへの管理者操作（事前のロール付与・MFA 強制等）ができない**点に留意が必要です。

Yes/No + 委譲範囲（ユーザー追加のみ / 全権委譲 / 段階的）でお答えいただけますと幸いです。

### 【Webhook 通知（基盤 → 御社の各アプリ）】 (B-405, 🟢)

**共通基盤側でユーザーイベントが発生した時、御社の各アプリへリアルタイム通知する必要**はございますか。これは **共通基盤 → 御社アプリ への push 通知**（SCIM とは方向が逆、[§FR-9.3.0](../proposal/fr/09-integration.md#fr-930-webhook-の役割と-scimjit-との違い) 参照）です。

典型シナリオ:
- `user.created`（JIT / SCIM / 手動いずれの契機でも発火）→ 御社アプリ DB の users テーブルに同期反映
- `user.deleted`（退職 SCIM 等）→ 御社アプリ側の関連リソース（ファイル所有権 / 履歴匿名化）クリーンアップ
- `mfa.enrolled` → Slack 通知 / 監査ログ記録
- `auth.compromised_credentials_detected` → SOC ダッシュボードへエスカレーション
- `session.revoked` → 御社アプリ側セッションキャッシュ無効化

Yes/No + 必要なイベント種別 + 連携先システム（内製アプリ / Slack / Datadog / SOC 等）でお答えいただけますと幸いです。
**目的**: イベント駆動連携の必要性判断、Cognito Lambda Trigger / Keycloak Event Listener の実装スコープ確定に必要な情報です。**Webhook 不要の場合**は、御社アプリ側で必要に応じてポーリング / SCIM 同期で代替する設計になります。

### 【新規作成時のデフォルト権限】 (B-401-2, 🟡)

**ユーザーレコードが共通基盤側で新規作成された時のデフォルト権限・ロール**についてご教示ください。本項は **JIT / SCIM / バルクインポート 全契機共通** の質問です。

具体的には以下 3 契機すべてで適用されます:
- **JIT**: フェデユーザーが初回ログインしてきた時、IdP アサーションの `groups` / `roles` クレームが**読み取れない / 該当値がない** 場合の fallback
- **SCIM**: SCIM ペイロードに `groups` 属性が**未指定** で push されてきた場合
- **バルクインポート**: CSV に `role` 列が**ない or 空** の行を投入した場合

業界推奨は **最小権限**（明示的にロールが付与されるまで何もできない、後から属性マッピングや管理者操作で上書き）です。これに従うか、別のレベル（例: "user" 基本ロール 自動付与）を採用するかお答えいただけますと幸いです。
**目的**: 新規ユーザー作成時の初期権限ポリシー設計、Cognito Pre Sign-up Lambda / Keycloak First Login Flow + SCIM Endpoint 実装に必要な情報です。**意図せぬ過剰権限付与を防ぐ**ためのセキュリティベースライン。

### 【ユーザー作成イベントの通知先】 (B-401-3, 🟢)

**共通基盤側でユーザーレコードが作成された時の通知先**をご教示ください。本項は **JIT / SCIM / バルクインポート 全契機共通** の質問です（B-401-2 と同じく、契機を問わず統一）。

通知先の選択肢:
- **CloudWatch Logs / Audit Log への記録のみ**（運用調査用）
- **SIEM（Splunk / Datadog 等）へストリーミング**（リアルタイム監査・不正検知用）
- **管理者メール通知**（初回 JIT 時に SOC や運用者に通知し、不審な作成を即座に気づける）
- **B-405 Webhook で外部アプリへ送る**（御社アプリと連動）
- 不要（基盤の標準ログのみ）

通知契機も併せて指定可能です（**JIT 時のみ通知 / SCIM 時のみ通知 / 全契機で通知**）。
**目的**: 監査ログ要件への適合、ユーザー作成の追跡性（誰がいつどの契機で作成されたか）、不正検知への利用判断に必要な情報です。特に **初回 JIT 通知** は、想定外の IdP からのフェデ流入を検知する重要な指標になります。

### 【同一テナント内ユーザー重複の想定】 (B-406, 🟡)

同一テナント内で同一人物が複数経路でアクセスする想定はございますか。具体的に想定されるシナリオ:
- **複数 IdP 接続**: Acme 社が Entra ID と HENNGE を併用、同じ人が両方からログインする
- **IdP 切替期間**: Okta → Entra への移行過渡期、同じ人が新旧両方で認証してくる
- **ローカル + フェデ併存**: 同じ email の人がローカルユーザー登録もしている
- **SCIM 先行作成 + JIT 後追い作成の競合**: SCIM で先に作られたレコードに、後で同じ人がフェデログインして JIT が走る
- **退職再入社**: 退職処理で削除されたユーザーが再入社で同じ email で復活する

あり（具体的な経路）/ なし、でお答えいただけますと幸いです。
**目的**: アカウント統合（リンク）設計の必要性判断、Cognito の `AdminLinkProviderForUser` / Keycloak の First Broker Login Flow の実装方針確定に必要な情報です。詳細は [§FR-2.2.1.A](../proposal/fr/02-federation.md) を参照。

### 【重複検出時の挙動】 (B-407, 🟡)

B-406 で「あり」の場合、重複検出時の挙動をご教示ください:
- 自動リンク（IdP 信頼前提、**乗っ取りリスクあり**）
- Email OTP 確認後リンク
- 既存パスワード再認証後リンク
- エラー停止（重複を許容しない）

**目的**: アカウントリンクの安全性確保。自動リンクは IdP の email クレームを信頼する設計のため、悪意のある IdP からの email アサーション流入で既存アカウント乗っ取りリスクが発生します。Email OTP / 再認証確認を推奨ベースラインとしています。

---

### 【重複検出の突合せキー】 (B-408, 🟡)

アカウント統合（リンク）時の突合せキーをご教示ください。
- email
- immutable な `sub` / `objectid`
- 雇用 ID（社員番号等）
- カスタム属性

**email のみは乗っ取りリスクが高い**ため、immutable な属性を主キーに、email を補助に使う設計を推奨しています。
**目的**: アカウントリンクの安全性確保、特に乗っ取り対策のための突合せキー設計に必要な情報です。

### 【IdP 切り替え時のユーザー連続性要件】 (B-410, 🟢)

顧客企業が IdP を切り替えた場合（例: Okta → Entra ID）、同一プロファイルを保つ要件はございますか。
あり / なし / 検討中、でお答えいただけますと幸いです。
**目的**: IdP 切替時の事前リンク戦略、もしくは管理者主導の手動マージ設計の必要性判断に必要な情報です。Yes の場合、SCIM 同期での事前リンク or 切替前の手動アカウント統合作業が運用フローに組み込まれます。
