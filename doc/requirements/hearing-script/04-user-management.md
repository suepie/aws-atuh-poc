### 【SCIM プロビジョニング】 (B-401, 🟡)

JIT（Just-in-Time）のみでユーザー作成を運用するか、SCIM 2.0 で IdP からの自動同期を併用するか、ご希望をご教示ください。
JIT のみ / SCIM 併用、もしくは想定退職フローの規模感（年間 / 月間の退職者数）でお答えいただけますと幸いです。
**目的**: SCIM 採用要否の判断、Cognito 採用時の SCIM 自前実装コスト評価（Cognito はネイティブ非対応、Lambda + DynamoDB 実装）、Keycloak SCIM プラグイン採用判断に必要な情報です。

---

### 【セルフサービス機能の範囲】 (B-402, 🟡)

ユーザーが自身で操作できる機能の範囲をご教示ください:
- パスワードリセット（メール経由）
- プロフィール属性編集
- MFA 要素の登録・解除
- アクティブセッションの確認・破棄
- 連携 IdP の追加・削除

**目的**: Keycloak Account Console / Cognito Hosted UI / 自前 UI の採用判断、必要な API エンドポイント設計、ユーザー UI 実装スコープに必要な情報です。

---

### 【バルクインポート方式】 (B-403, 🟢)

既存ユーザーの初期投入方法をご教示ください。
- バルクインポート（CSV 一括投入）
- SCIM 経由
- JIT 任せ（初回ログイン時に作成）

規模感（投入予定ユーザー総数）も併せてお答えいただけますと幸いです。
**目的**: 初期移行設計、Cognito の AdminCreateUser API レート制限（50 RPS）への対応設計、Keycloak のインポート機能利用判断に必要な情報です。

---

### 【テナント管理者の委譲】 (B-404, 🟢)

顧客企業の管理者が、自社テナント内のユーザー / ロール / MFA ポリシーを自律管理する形式は必要でしょうか。
Yes/No + 委譲範囲（ユーザー追加のみ / 全権委譲 / 段階的）でお答えいただけますと幸いです。
**目的**: 委譲管理（Delegated Administration）採用判断。**Yes の場合、Keycloak の Realm-level Admin Role が標準対応、Cognito は AWS IAM 設計が複雑化**するため、プラットフォーム選定に影響します。

---

### 【Webhook 通知】 (B-405, 🟢)

`user.created` / `user.deleted` 等のユーザーライフサイクルイベントを、外部システムに Webhook で通知する必要はございますか。
Yes/No + 連携先システム名でお答えいただけますと幸いです。
**目的**: イベント駆動連携の必要性判断、Cognito Lambda Trigger / Keycloak Event Listener の実装スコープ確定に必要な情報です。

---

### 【JIT デフォルト権限レベル】 (B-401-2, 🟡)

JIT（フェデユーザー初回ログイン時の自動作成）におけるデフォルト権限についてご教示ください。
業界推奨は**最小権限**（後から属性マッピングでロール上書き）です。これに従うか、別のレベルを採用するかお答えいただけますと幸いです。
**目的**: JIT 時の初期権限ポリシー設計、Cognito Pre Sign-up Lambda / Keycloak First Login Flow の実装に必要な情報です。

---

### 【JIT 生成イベントの通知先】 (B-401-3, 🟢)

JIT による自動ユーザー生成イベントの通知先をご教示ください。
CloudWatch / SIEM（Splunk 等） / メール通知 / 不要、のいずれかでお答えいただけますと幸いです。
**目的**: 監査ログ要件への適合、JIT 生成の追跡性（誰がいつ自動生成されたか）、不正検知への利用判断に必要な情報です。

---

### 【同一テナント内ユーザー重複の想定】 (B-406, 🟡)

同一テナント内で同一人物が複数経路でアクセスする想定はございますか。
具体例: 複数 IdP 接続（Acme = Entra ID + HENNGE） / IdP 切替期間 / ローカル + フェデ併存 / SCIM + JIT 競合 / 退職再入社 等。
あり（経路の例）/ なし、でお答えいただけますと幸いです。
**目的**: アカウント統合（リンク）設計の必要性判断、Cognito の `AdminLinkProviderForUser` / Keycloak の First Broker Login Flow の実装方針確定に必要な情報です。詳細は [§FR-2.2.1.A](../proposal/fr/02-federation.md) を参照。

---

### 【重複検出時の挙動】 (B-407, 🟡)

B-406 で「あり」の場合、重複検出時の挙動をご教示ください:
- 自動リンク（IdP 信頼前提、**乗っ取りリスクあり**）
- Email OTP 確認後リンク
- 既存パスワード再認証後リンク
- エラー停止（重複を許容しない）

**目的**: アカウントリンクの安全性確保。自動リンクは IdP の email クレームを信頼する設計のため、悪意のある IdP からの email アサーション流入で既存アカウント乗っ取りリスクが発生します。Email OTP / 再認証確認を推奨ベースラインとしています。

---

### 【重複検出の突合せキー】 (B-408, 🟡)

アカウント統合（リンク）時の突合せキーをご教示ください。
- email
- immutable な `sub` / `objectid`
- 雇用 ID（社員番号等）
- カスタム属性

**email のみは乗っ取りリスクが高い**ため、immutable な属性を主キーに、email を補助に使う設計を推奨しています。
**目的**: アカウントリンクの安全性確保、特に乗っ取り対策のための突合せキー設計に必要な情報です。

---

### 【アカウントリンクのトリガー】 (B-409, 🟢)

アカウントリンクをいつ・誰がトリガーするかご教示ください:
- 管理者主導（基盤運用 / テナント管理者）
- ユーザー主導（Account Console でユーザー自身が紐付け）
- 自動（信頼できる IdP 条件下で）

**目的**: アカウントリンクの運用フロー設計、UI 提供範囲の確定、Keycloak Account Console の利用判断に必要な情報です。

---

### 【IdP 切り替え時のユーザー連続性要件】 (B-410, 🟢)

顧客企業が IdP を切り替えた場合（例: Okta → Entra ID）、同一プロファイルを保つ要件はございますか。
あり / なし / 検討中、でお答えいただけますと幸いです。
**目的**: IdP 切替時の事前リンク戦略、もしくは管理者主導の手動マージ設計の必要性判断に必要な情報です。Yes の場合、SCIM 同期での事前リンク or 切替前の手動アカウント統合作業が運用フローに組み込まれます。
