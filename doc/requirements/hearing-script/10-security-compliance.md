### 【FIPS 140-2 認定の必要性】 (C-201, 🔥)

取得・準拠が必要な認証 / 規格をご教示ください:

**目的**: 各認証への適合設計、Cognito の AWS 認証取得状況 / Keycloak での自前認証取得の判断、監査ログ要件・暗号化要件・運用フロー設計に必要な情報です。

### 【監査ログ保存期間】 (C-203, 🔥)

監査ログの保存期間をご教示ください。**ログ種別と適用法規によって要件が大きく異なる**ため、以下の観点で整理いただけますと幸いです。

**対象ログ種別**:
- 認証イベントログ（ログイン成功 / 失敗 / MFA / ログアウト）
- 管理操作ログ（ユーザー CRUD / 設定変更 / 権限変更）
- 設定変更履歴（IdP 追加 / Realm 設定変更）

**法定要件の参考値**:
- SOC 2: 1 年以上
- PCI DSS: 1 年（うち 3 ヶ月は即時アクセス可能）
- **HIPAA: 6 年**（医療情報）
- **FFIEC: 5-7 年**（金融）
- GDPR: データ主体権利応答可能期間
- 業界独自規制（FISC 等）

具体的な期間（ログ種別ごとに違う場合は分けて）でお答えいただけますと幸いです。
**目的**: CloudWatch Logs / Keycloak Event Listener + 外部ストレージの設計、長期保存ストレージ（S3 Glacier）の容量見積、コンプライアンス要件への適合性確認に必要な情報です。

### 【パスワードポリシー（基本）】 (C-204, 🟡)

**ローカルユーザー**（共通基盤側に直接パスワード登録するユーザー）のパスワードポリシー要件をご教示ください。

**対象ユーザー**（[§FR-1.2.0.0 利用者カテゴリ](../proposal/fr/01-auth.md#fr-1200-ローカルユーザーとは何か--利用者カテゴリ別の分析) と連動）:
- P-1 基盤運用管理者の Break Glass 用ローカル
- P-2 テナント管理者（御社が IdP を持たない場合）
- P-4 IdP なし顧客従業員
- P-6 B2C エンドユーザー（該当時）

**御社 IdP 経由でログインするフェデユーザー（P-3）は、御社 IdP 側のポリシーが適用される**ため本項の対象外です。

確認事項:
- **最小長**（例: 8 / 12 / 14 文字）
- **複雑性**（大文字 / 小文字 / 数字 / 記号の組み合わせ。**NIST SP 800-63B Rev 4 では複雑性要件は推奨されなくなった**点に注意）
- **適用粒度**: 基盤共通のポリシーで全顧客統一 / 顧客ごとに別ポリシー許容

**目的**: Cognito Password Policy / Keycloak Password Policies の設定値確定、NIST SP 800-63B Rev 4 準拠の判断、顧客別ポリシー対応の必要性確認に必要な情報です。

### 【NIST SP 800-63B Rev 4 準拠方針】 (C-204-3, 🟡)

NIST SP 800-63B Rev 4（2024 公開）への準拠方針をご教示ください:
- 完全準拠（複雑性要件削除 / 90 日ローテーション非推奨 / 侵害検出推奨）
- 部分採用（特定項目のみ）
- 不要（既存ポリシー維持）

**目的**: パスワードポリシー設計、Cognito Plus ティア（侵害クレデンシャル検出）の必要性、Keycloak の HIBP プラグイン採用判断に必要な情報です。

### 【既存パスワードハッシュの移行】 (C-204-4, 🟡)

既存ユーザーのパスワードハッシュ形式と、本基盤への持ち越し希望をご教示ください:
- ハッシュ形式（bcrypt / scrypt / Argon2 / PBKDF2 / MD5・SHA-1 等のレガシー）
- 持ち越し有無（あれば再ハッシュ不要、なければ全ユーザーパスワードリセット）
- 対象件数

**目的**: 移行戦略の確定。Cognito は Pre Sign-up Lambda で限定的にカスタムハッシュ対応、Keycloak は Password Hashing Provider のカスタマイズで対応可能。レガシーハッシュ（MD5 / SHA-1）の場合、セキュリティ観点で全リセット推奨となります。

### 【既存システムで独自ローカル認証を持つアプリの有無】 (C-204-5, 🔥)

> **本問は [マスター表 C](01-auth-flow.md#マスター表-c-御社アプリシステム構成リスト) 列 T（既存ローカル認証 + 移行方針）に統合済**です。該当アプリで列 T を **N（なし）/ M1（段階移行）/ M2（並行稼働）/ M3（即時切替）/ M4（維持）** から選択してください。本ページの本項は**該当判定の参考情報**として残しています。

アプリ独自の Login UI + ユーザー DB + パスワード管理を持つ既存システムが**1 つでも該当する場合**、表 C で該当アプリの列 T で M1〜M4 を選択し、補足列に件数を記入してください。

**目的**: 本基盤は **A 案: 共通基盤集約**が前提です。C 案ハイブリッド（一部アプリのみ独自認証維持）は移行期限定で許容します。「あり（=表 C 列 T が M1〜M4）」の場合、[§FR-1.2.0](../proposal/fr/01-auth.md) の移行戦略策定が必要となります。

### 【アカウントロック詳細設定】 (C-205, 🟡)

アカウントロックの設定要件をご教示ください:
- 自動 BF（Brute-Force）保護で十分（Cognito 標準 / Keycloak Brute Force Detection）

漏洩パスワード（既知のリーク DB に含まれるパスワード）の自動検知・拒否が必要でしょうか。
有無でお答えいただけますと幸いです。
**目的**: NIST 推奨機能。**Yes の場合、Cognito Plus ティア（Compromised Credentials Detection）または Keycloak + HIBP（Have I Been Pwned）プラグイン採用**となります。

### 【トークン TTL】 (C-206, 🟡)

**共通基盤が発行する**各トークン（御社の各アプリが受け取る JWT）の有効期限をご希望をご教示ください:
- **Access Token**（業界推奨: 15〜60 分。**短いほど侵害ウィンドウが小さい**が、Refresh 頻度が上がる）
- **Refresh Token**（業界推奨: 30 日、Token Rotation 採用時）
- **ID Token**（業界推奨: 15〜60 分）

**トレードオフ**:
- **短 TTL（5-15 分）**: 侵害ウィンドウ小、退職者が手元のトークンで API アクセスし続けられる時間が短い。ただし Refresh 頻度増加で API ロード・UX 悪化、Cognito の M2M クライアント 150 RPS 上限に注意
- **長 TTL（60 分）**: UX 良好、API ロード低い。ただし退職時の侵害ウィンドウが長くなる → 別途 **B-704 Access Token Revocation** で補完が必要

→ **短 TTL で吸収する** か **長 TTL + Token Revocation で対処する** か、設計方針の選択になります。退職反映 SLA（**B-605-3**）と連動してご判断ください。Revocation 採用の判断軸・実装コスト比較は **[マスター表 C 補足 2 K8](01-auth-flow.md#k8-access-token-即時-revocation--両プラットフォームで個別-revoke-不可)** を参照。

**目的**: トークンライフサイクル設計、退職時の即時遮断要件との整合性、UX（再ログイン頻度）とセキュリティ（漏洩時の被害最小化）のバランス調整に必要な情報です。

### 【トークン失効要件】 (C-207, 🟡)

> **本問は [マスター表 C](01-auth-flow.md#マスター表-c-御社アプリシステム構成リスト) 列 S K8（Access Token 即時 Revocation）に統合済**です。旧 B-704 と完全に同じ問いです。該当アプリで列 S K8 を☑してください。

トークンの即時無効化が必要なアプリが**1 つでも該当する場合**、表 C で該当アプリの列 S K8 を☑してください。

**詳細解説**（短 TTL で吸収 vs 長 TTL + Revocation のトレードオフ表 / 該当判定 / Cognito 自前 vs Keycloak Introspection の実装/運用コスト比較 / [C-206 トークン TTL](#トークン-ttl-c-206-)・[B-605-3 退職反映 SLA](../hearing-checklist.md)・[B-401 SCIM 採否](04-user-management.md) との連動）は **[マスター表 C 補足 2 K8](01-auth-flow.md#k8-access-token-即時-revocation--両プラットフォームで個別-revoke-不可)** を参照ください。

**目的**: Token Revocation 実装方式の確定。両プラットフォームとも Access Token の個別 revoke は不可で、Cognito は `origin_jti` 自前実装、Keycloak は Token Introspection 標準提供で対応します。

### 【目標 NIST AAL レベル】 (C-210, 🔥)

認証保証レベル（Authenticator Assurance Level）の目標をご教示ください:
- **AAL1**: パスワードのみ（推奨されない）
- **AAL2**: MFA 必須（**推奨**、業務系 SaaS の標準）
- **AAL3**: Phishing-resistant MFA 必須（Hardware Key / Passkey）

**目的**: 認証強度の根本基準、MFA 要素の選定、IdP の AAL 評価、ステップアップ MFA 設計に必要な情報です。**プラットフォーム選定にも影響**します（AAL3 必須で Keycloak の標準 ACR-to-LoA 機能が有利）。

### 【Phishing-resistant MFA 採用方針】 (C-211, 🟡)

Passkey / FIDO2 を Must / Should / Could のどの位置付けで採用するかご教示ください。
業界調査では 87% の Enterprise が deploy / pilot 中であり、NIST AAL2 / AAL3 整合の標準的選択肢です。
**目的**: MFA 方式選定、Cognito Essentials+ ティア（2024-11〜 WebAuthn 対応）/ Keycloak 標準対応の活用判断に必要な情報です。

### 【条件付き MFA の判定軸】 (C-214, 🟡)

リスクベース MFA（条件付き MFA）の判定軸として何を採用するかご教示ください:
- IP アドレス（社内 IP のみスキップ等）
- 地理的位置（国 / 地域）
- デバイス（既知端末 / 未登録端末）
- 時間帯（業務時間内のみスキップ等）
- 行動パターン（異常検知）
- これらの組み合わせ

**目的**: Cognito Plus ティア（Adaptive Authentication）/ Keycloak の Adaptive Authentication 採用判断、条件付き MFA の実装スコープ確定に必要な情報です。

業界デフォルトは 30 日、範囲は 0〜90 日が一般的です。
**目的**: UX とセキュリティのバランス調整、Cookie 寿命設計、端末紛失時のリスク評価に必要な情報です。

### 【ステップアップ認証（RFC 9470）の要否】 (C-216, 🟡)

高セキュリティ操作（決済 / 管理画面 / 大量データ出力等）で、動的に AAL を引き上げる仕組みが必要でしょうか。
有無 + 対象操作でお答えいただけますと幸いです。
**目的**: RFC 9470 採用要否の判断。**Yes の場合、宣言的実装なら Keycloak（ACR to LoA Mapping 標準）、Cognito は Custom Auth Challenge Lambda で自前実装**となります。詳細は [§FR-3.3](../proposal/fr/03-mfa.md) を参照。

### 【CAEP / 継続的アクセス評価（将来発展）】 (C-217, 🟢)

リアルタイム deprovision / デバイス侵害時の即時遮断 / リスクシグナル伝播の仕組みは必要でしょうか。
- 必須（現時点で要求）
- 将来検討
- 不要

**目的**: CAEP（OpenID Continuous Access Evaluation Profile）採用要否の判断。**現時点で Cognito / Keycloak とも未対応**のため、必須なら別途追加実装または Microsoft Entra との統合検討が必要となります。将来発展経路として位置付けるのが現実解です。
