## 用語の前提 — 「認可」という言葉の 2 つの意味（事前にすり合わせ）

> **本セクションで扱う質問はすべて「意味 B の認可」と「意味 A の認可フロー」の関係を前提**としています。混同しやすいので、本論に入る前に整理させてください。

OAuth / OIDC 周辺で「認可」は 2 つの意味で使われ、混同されがちです:

| 意味 | 何の話か | 担当 |
|---|---|---|
| **意味 A: 認可フレームワーク**（OAuth 2.0 そのもの）| **Token をどう発行するか**（フロー / プロトコル）| **本基盤**（Authorization Server）|
| **意味 B: 認可判定**（リソース保護）| **alice は /expense/123 を編集できるか?** という業務判定 | **御社の各アプリ**（Resource Server）|

### 本基盤のスタンス（[§FR-6.0.A](../proposal/fr/06-authz.md) より）

> **「認可は御社アプリ側」スタンス = 意味 B の認可は御社アプリの責務**。本基盤は「**認証 + 最小限のクレーム発行**」までを担当します。

### よくある誤解とその訂正

| 誤解 | 訂正 |
|---|---|
| 「認可はアプリ側だから、本基盤は Token 発行だけしてれば良い」 | **半分正解**。意味 B の判定はアプリ側だが、**意味 A の Token 発行制御**（aud / scope / 発行先）は本基盤が責任を持つ |
| 「Cognito / Keycloak はクレームを整えるだけ」 | **半分正解**。認可判定（意味 B）はしないが、**「どの Token をどのアプリに、どんな権限で発行するか」を制御**している（意味 A）|
| 「本基盤が Token Exchange できないとアプリ側で何が困るか分からない」 | **困る**。Token Exchange なしだとアプリ側で「`aud` チェックを甘くする」「ユーザー文脈を捨てる」等の妥協が必要 → **アプリ側認可がルーズ化** |

### Token Exchange が必要になるシナリオ（マイクロサービス OBO）

> **本セクションのリッチコンテンツ（シーケンス図 / 代替策表 / 判定フローチャート / 業務シナリオ 7 軸）は [マスター表 C 補足 2 K1](01-auth-flow.md#k1-token-exchangerfc-8693--cognito-非対応) に統合・拡充済**です。Token Exchange の典型シナリオ・代替策の妥協・判定フロー・コンテンツ全体は補足 2 K1 を参照ください。
>
> 本セクションは **「意味 A の認可フロー vs 意味 B の認可判定」の用語整理**（前置き）と、**B-304 業務シナリオ質問**（K1 への該当判定）として残ります。

### 【必須クレーム】 (B-301, 🔥)

各システム（アプリ）が JWT に必要とする属性をご教示ください。
具体的な属性リスト（例: `sub`, `tenant_id`, `email`, `roles`, `groups`, カスタム属性等）でお答えいただけますと幸いです。
**目的**: 本基盤が発行する統一クレーム形式の確定、Cognito の Pre Token Lambda V2 / Keycloak の Protocol Mapper の設計、JWT サイズ最適化に必要な情報です。最小クレーム設計（`iss` / `sub` / `aud` / `azp` / `tenant_id` / `exp` / `iat` のみ）を業界標準として推奨しています（[§FR-6.1.A](../proposal/fr/06-authz.md)）。

### 【認可粒度】 (B-302, 🟡)

各システムの認可判定をどの粒度で行うかの設計方針をご教示ください。
- ロール単位（admin / user / viewer 等）
- リソース単位（特定ドキュメント / 特定レコード）
- アクション単位（read / write / delete）
- 組み合わせ

**目的**: 各アプリの認可設計パターン（A: アプリ DB / B: RBAC / C: ABAC / D: 外部 PDP）の選定支援、本基盤の JWT クレーム設計の判断、Keycloak Authorization Services 採用要否の判断に必要な情報です。

### 【細粒度認可（UMA 2.0）の必要性】 (B-303, 🟡)

> **本問は [マスター表 C](01-auth-flow.md#マスター表-c-御社アプリシステム構成リスト) 列 S K6（UMA 2.0 細粒度認可）に統合済**です。該当アプリで列 S K6 を☑してください。本ページの本項は**該当判定の参考情報**として残しています。

「ドキュメント X を user A だけが閲覧可、user B は編集可」のようなリソース所有者ベースの細粒度認可が必要なアプリが**1 つでも該当する場合**、表 C で該当アプリの列 S K6 を☑してください。

**目的**: UMA 2.0（User-Managed Access）採用要否の判断。**☑の場合（=表 C 列 S K6 がいずれかのアプリで含まれる）、Cognito ネイティブ非対応のため Keycloak Authorization Services 必須**、もしくは外部 PDP（Amazon Verified Permissions + Cedar / OPA / OpenFGA）の採用が必要となります。

### 【API 間トークンリレー / ユーザー文脈伝播】 (B-304, 🟡)

> **本問は [マスター表 C](01-auth-flow.md#マスター表-c-御社アプリシステム構成リスト) 列 S K1（Token Exchange）に統合済**です。該当アプリで列 S K1 を☑してください。本ページの本項は**該当判定の参考情報**として残しています。

マイクロサービス間呼び出しで「**誰の操作か**」を伝播する必要があるアプリが**1 つでも該当する場合**、表 C で該当アプリの列 S K1 を☑してください。

**該当判定の業務シナリオ 7 軸**（いずれか Yes → 列 S K1 ☑）:
- ① マイクロサービス構成を採用している
- ② サービス A → B 内部呼び出しがある
- ③ B 側のログでエンドユーザー追跡が必須
- ④ サービス別の個別権限チェックが必要
- ⑤ scope 縮小（最小権限の段階的適用）が必要
- ⑥ 外部システムへの代理操作（On-Behalf-Of）
- ⑦ コンプライアンス要件（個人情報アクセスの追跡 等）

**詳細解説**（シーケンス図 / 代替策の妥協 3 つ / 判定フローチャート）は **[マスター表 C 補足 2 K1](01-auth-flow.md#k1-token-exchangerfc-8693--cognito-非対応)** を参照ください。

**目的**: Token Exchange（RFC 8693）採用要否の判断。**☑の場合（=表 C 列 S K1 がいずれかのアプリで含まれる）、Cognito ネイティブ非対応のため Keycloak 必須化**となります。詳細判定フローは [§FR-6.3.4](../proposal/fr/06-authz.md) を参照してください。

### 【テナント分離粒度】 (B-306, 🔥)

> **問いの位置づけ**: マルチテナント環境における **認証境界（Keycloak Realm / Cognito User Pool）と業務データ分離の粒度** を確定する。「全顧客で 1 つの認証境界を共有 + tenant_id クレームで識別」（L2 論理分離、業界標準）vs 「顧客ごとに Realm/Pool を分離」（L3 物理分離、規制業種向け）の選択。
> **回答で決まること**: ①認証境界数（1 つの Realm/Pool vs 顧客数だけの Realm/Pool）/ ②Identity Broker パターン採用可否（L3 だと崩壊）/ ③SSO 範囲（L2 = テナント内アプリ間 / L3 = 同一顧客 Realm 内）/ ④N×M Client 登録の有無 / ⑤Cognito Custom Domain 4 個 Hard Limit 抵触 / ⑥認可設計（JWT クレーム vs Realm 識別）/ ⑦業務データ DB の分離方式。

#### 前提：用語の整理（Realm ≈ User Pool は同じ概念）

| 概念 | Keycloak | Cognito | 何か |
|---|---|---|---|
| **認証境界**（tenancy boundary）| **Realm** | **User Pool** | ユーザー / IdP / クライアントを内包する独立した認証単位 |
| **アプリ登録単位** | Client | App Client | 個別アプリ（expense-app、payment-app 等）の設定 |
| **SSO 範囲** | Realm 内のクライアント間で自動成立 | Pool 内の App Client 間で自動成立 | **境界をまたぐ SSO は自動成立しない** |

→ **Realm と User Pool は同列に語れる**（両者は「認証境界」という同じ概念）。

#### よくある誤解への回答

**Q1: Realm をアプリ単位（システム単位）で分けるのは?**
**A**: ❌ **非推奨**。各アプリで独立した Realm を作ると **アプリ間 SSO が完全に失われる**（Realm 境界をまたいだ SSO は成立しないため）。「expense ログイン → payment 再ログイン要求」となり、SSO 基盤の意味がなくなる。

**Q2: Realm を顧客（IdP）単位で分けることは可能か?**
**A**: ✅ **可能**。それが **L3 物理分離**。同一 Realm 内のアプリ間で SSO が成立するため、**「顧客 acme のユーザーが acme の全アプリで SSO」は実現できる**。ただしデメリットあり（後述）。

#### 比較イメージ（L1 / L2 / L3）

| 観点 | **L1 完全集約** | **L2 論理分離（標準・業界推奨）** | **L3 物理分離** |
|---|---|---|---|
| **構成** | 1 Realm/Pool、外部 IdP なし、ローカルユーザーのみ | **1 Realm/Pool**、複数 IdP 登録、tenant_id クレームで識別 | **N Realm/Pool**（顧客数分）|
| **SSO 範囲** | 共通基盤内全アプリ | **同一顧客のアプリ間で自動成立** | 同一 Realm 内 = 同一顧客のアプリ間で成立 |
| **顧客追加コスト** | n/a（IdP なし）| 新 IdP 連携 1 回 | **新 Realm + 全アプリ Client を再登録**（M 回）|
| **アプリ追加コスト** | 1 Client 登録 | 1 Client 登録（全顧客で利用可）| **N 顧客 × 1 Client = N 回登録** |
| **Cognito Custom Domain** | 共通 1 つ | 共通 1 つ | **顧客数分必要、4 個 Hard Limit 抵触** |
| **Identity Broker パターン** | ✅ 標準（顧客 0）| ✅ **標準形** | ⚠ **崩壊**（N 個の Broker = もはや Hub ではない）|
| **業務データ DB 分離** | 共通 DB | 共通 DB + tenant_id（推奨）or DB 分離 | 顧客別 DB + 別 AWS アカウント等の物理分離が伴うことが多い |
| **採用例** | 社内ツール / B2C | **Slack / Notion / Linear / Box / Auth0 / Entra B2B** | 規制業種専用 SaaS、AWS GovCloud 様の物理分離 |
| **本基盤の標準扱い** | 顧客 IdP なしの場合 | **標準採用**（業界推奨）| **例外的に物理分離契約・規制要件の顧客のみ採用**（追加コスト）|

→ **業界標準は L2 論理分離**。L3 は **コンプラ要件 / 顧客契約で物理分離必須**な場合のみ採用。L1〜L6 の詳細グラデーションは [§C-1.4 物理分離レベルと Broker パターンの関係](../proposal/common/01-architecture.md#c-14-物理分離レベルと-broker-パターンの関係)。

#### 規模別の影響（顧客企業数 = テナント数 = IdP 接続数）

> **MAU とは別軸**で、**顧客企業数（テナント数）**が **Cognito Hard Limit に直接抵触**します。**A-15 顧客企業数**（[00-common.md](00-common.md)）と連動して、L2 維持可否・プラットフォーム選定を決定。

| 顧客企業数 | L2 単一 Pool/Realm 維持可否 | Cognito 影響 | Keycloak 影響 | 本基盤の推奨対応 |
|---:|---|---|---|---|
| **〜100 社** | ✅ 維持可（両プラットフォーム）| なし | なし | 標準 L2 |
| **〜500 社** | ⚠ Cognito は SAML 顧客多数で Pool 分割要 | SAML IdP 〜100 上限 | なし | Keycloak 推奨、Cognito なら 5 Pool |
| **〜1500 社** | ❌ Cognito 単一 Pool 不可 | **15 Pool 必須 + Custom Domain 4/region 抵触** | Organization 機能必要 + DB チューニング | **Keycloak 強推奨** |
| **〜3000 社** | ❌ Cognito 30 Pool（運用爆発）| 30 Pool、運用負荷線形増加 | 単一 Realm or 2-3 分割、性能監視重要 | **Keycloak 強推奨** |
| **〜10000 社** | ❌ Cognito 事実上不可 | - | Realm 分割 + 専任 SRE | Keycloak（複数 Realm 分割）|

→ **1500-3000 顧客企業規模では Keycloak を強推奨**。Cognito 採用時は **「変形 L2」**（cohort 別 Pool 分割、Lambda@Edge による Path ルーティング、multi-issuer Lambda Authorizer）となり運用負荷が顕著に重い。詳細は [§C-1.5 規模スケーリング戦略](../proposal/common/01-architecture.md#c-15-規模スケーリング戦略1500-3000-顧客企業)。

#### なぜこれを今聞くのか

**テナント分離粒度は認証基盤の根本構造**で、後から変更すると **Realm/Pool 統廃合 = ユーザー DB 移行 = 全顧客の再ログイン強制**という重大インシデント級の作業になります。設計時に確定が必須。

特に **L3 物理分離は「契約や規制で物理分離必須」な場合のみ**選ぶべきで、安易な L3 採用は **N×M Client 登録 / Custom Domain 上限 / Identity Broker パターン崩壊 / 設定ドリフト**を招きます。「物理分離してほしい」という顧客要望の **真のニーズ**（コンプラ / 監査 / 暗号鍵分離 等）を掘り下げ、L2 + 認可で満たせるかをまず検討するのが業界標準。

#### 質問

マルチテナント環境におけるテナント分離方式をご教示ください。

**認証境界（Realm / User Pool）レベル**:
- **L1 完全集約**: 顧客 IdP なし、共通基盤ローカルユーザーのみ（B2C / 社内ツール）
- **L2 論理分離（推奨）**: 単一 Realm/Pool + 複数 IdP + `tenant_id` クレームで識別
- **L3 物理分離**: 顧客ごとに別 Realm/Pool（追加コスト、Identity Broker パターン崩壊、規制業種のみ）

**業務データ DB レベル**（並行して確認）:
- **共有 DB + tenant_id** で識別（L2 と組み合わせる業界標準）
- **DB 分離**（顧客ごとに別 DB、L2 でも採用可）
- **AWS アカウント分離**（L3 に近い、規制業種）
