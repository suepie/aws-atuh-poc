## Part 1: 集約するかしないか

### スライド 2: 2 つの方針 - 集約する / しない

**タイトル**: まずは「そもそも集約するか」から

🎯 **このスライドで何を確認したいか**: 「集約 / 個別」は独立した根本判断であることの共有

📝 **リード文**:
> データ活用基盤には **2 つの根本的な方針**があります。**中央に集約する方針**（各アプリのデータを横断で見られるようにする）と、**集約しない方針**（各アプリで独立してデータ活用する）です。どちらが良いかは、**ビジネス目的・期待価値・コスト・運用**の 4 観点で総合的に判断します。

📊 **表**: 2 方針の一言比較

| 観点 | **集約する方針**（A案 or B案）| **集約しない方針**（C案 個別型）|
|---|---|---|
| 全社横断で見える | ⭕ 可能 | ❌ 事実上不可 |
| 各アプリの独立性 | △ 中央依存あり | ⭕ 完全独立 |
| データを持つ場所 | 1 箇所（分散 or 中央）| 各アプリ個別 |
| 中央組織 | 必要（2-5 名以上）| 不要 |
| 初期投資 | 中央基盤構築（工数・アカウント設計）| 各アプリで独立に構築 |
| 月額運用コスト（10 アプリ想定）| ~$1,143-1,253 | ~$5,000-8,000（アプリ数で線形増加）|
| 障害影響範囲 | 中央障害で全社影響 | 1 アプリに閉じる |
| **業界通称** | 「データレイク」「データメッシュ」| 「独立運用」「サイロ型」 |

💡 **お伝えしたいこと**:
- 「集約する方針」は 業界で主流になりつつありますが、**必ずしも常に正解ではありません**
- **横断で見たい要件が明確にある**、かつ **投資対効果が見合う** 場合のみ推奨
- 「集約しない方針」は **アプリごとに自足**する伝統的な形、無理に集約しないほうが良いケースもあります
- **これから 2 スライド（目的と価値 → 総合比較）でどちらが弊社に合うかを検討**します

⚠ **本日の流れ**: スライド 3-4 で「集約するか / しないか」を暫定判断 → 集約するなら Part 2 へ進む

### スライド 3: 集約する目的と期待するビジネス価値

**タイトル**: 集約する目的は? 期待するビジネス価値は?

🎯 **このスライドで何を確認したいか**: 集約の**ビジネス的な意義**の有無を経営・事業視点で確認

📝 **リード文**:
> 集約するかどうかは、**経営・事業にどのような価値をもたらすか**で決まります。以下 5 つのビジネス目的について、**実際にご要望・関心があるか**、また**期待できる価値の大きさ**をご確認ください。

📊 **表 3.1**: 5 つのビジネス目的 と 期待価値

| # | ビジネス目的 | 具体的な問い | 期待できる価値（例）| ご関心度 |
|---|---|---|---|:---:|
| ① | **経営: 全社 KPI の一元把握** | 「今月の全社売上・解約・粗利を 1 画面で見たい」 | 意思決定 3-6 ヶ月早期化、経営会議の生産性向上 | ⬜ 高 / 中 / 低 |
| ② | **CS: 顧客の全 SaaS 横断把握** | 「1 顧客が複数 SaaS を使っている場合、活性度を横断で見て解約予兆を検知したい」 | 解約率 1-3% 改善（LTV 向上）、CS 提案精度向上 | ⬜ 高 / 中 / 低 |
| ③ | **マーケ: 全 SaaS の獲得ファネル横断** | 「どの入口から入って、どの SaaS に育つかのパスを把握したい」 | CPA 10-20% 改善、マーケ ROI 向上 | ⬜ 高 / 中 / 低 |
| ④ | **会計: 全 SaaS 売上合算・原価分析** | 「月次売上・粗利・原価を全社集計、事業別損益を早く出したい」 | 月次締め 2 営業日短縮、事業別収支の見える化 | ⬜ 高 / 中 / 低 |
| ⑤ | **開発・SRE: 全 SaaS の品質メトリクス横断** | 「エラー率・応答時間・利用ログを横断で見て、共通の品質改善を進めたい」 | 障害検知 MTTR 半減、共通不具合の早期発見 | ⬜ 高 / 中 / 低 |

📊 **表 3.2**: 期待価値の金額換算（叩き台、経営層と議論）

| 目的 | 期待価値（金額 / 年） | 算出根拠（例） |
|---|---:|---|
| ① 経営 | ⬜ | 意思決定早期化 → 施策の効果拡大 |
| ② CS | ⬜ | 解約率 1% 改善 → LTV × 顧客数 × 1% |
| ③ マーケ | ⬜ | CPA 10% 改善 → マーケ予算 × 10% |
| ④ 会計 | ⬜ | 月次締め工数削減 → 人月 × 単価 × 12 |
| ⑤ 開発 | ⬜ | 障害対応工数削減、機会損失削減 |
| **合計期待価値** | ⬜ | vs 月額コスト $1,143-1,253 = 年 $14K-15K |

💡 **想定シナリオ**:

| ご回答パターン | 判断 | 次のアクション |
|---|---|---|
| **3 個以上「高」** + 期待価値 > 年 $50K | 集約するビジネス価値 十分 | スライド 4 の総合比較へ、集約前提で議論 |
| 1-2 個「高」+ 期待価値 = $15-50K | 微妙、要議論 | スライド 4 でコスト精査、経営持ち帰り |
| **すべて「低」** or 期待価値 < 月コスト | 集約するビジネス価値 なし | **C案 個別型 に確定**、以降スキップ |

⚠ **回答内容が何を左右するか**:
- **明確な価値がある → 集約方針**、スライド 4 でコストと総合比較
- **価値が薄い or 見えない → C案 個別型 で確定**、Part 2 スキップ

### スライド 4-1: 総合比較 A. ビジネス効果

**タイトル**: 総合比較 A - ビジネス効果の観点

🎯 **このスライドで何を確認したいか**: **ビジネス効果**の観点で 集約 / 個別 を比較

📝 **リード文**:
> スライド 3 で確認したビジネス目的が、集約する / しない でどの程度実現できるかを比較します。**横断分析の可否**が最大のポイントです。

📊 **表 4-1**: ビジネス効果 の比較

| 観点 | 集約する方針（A案 / B案）| 集約しない方針（C案）|
|---|---|---|
| 全社横断分析（経営 KPI、CS 予兆、マーケ ファネル 等）| ⭕ 可能 | ❌ 不可 |
| 個別最適化・アプリ内 UX | △ 中央標準化に寄る | ⭕ 各アプリ自由 |
| 期待価値の実現性 | 5 つの目的（スライド 3）を全て実現可 | ①②③④⑤ 全て実現不可 or 各アプリ内のみ |

💡 **想定シナリオ**:

| ご判断 | 傾向 |
|---|---|
| 全社横断分析が必要 | **集約が有利** |
| 個別最適化を重視 | 集約しないほうが自由 |
| 各アプリ内で完結する分析で十分 | C案 で成立 |

⚠ **この観点の判断への影響**: **横断要件が明確なら集約、なければ C案 有利**

### スライド 4-2: 総合比較 B. コスト

**タイトル**: 総合比較 B - コストの観点

🎯 **このスライドで何を確認したいか**: **コスト**の観点で 集約 / 個別 を比較

📝 **リード文**:
> **初期投資・月額運用・年間コスト**の 3 段階で比較します。集約は初期投資が大きく月額は小さい、個別は逆に初期は小さいが月額が線形増加、という構造です。

📊 **表 4-2**: コスト の比較（10 アプリ想定）

| 観点 | 集約する方針（A案 / B案）| 集約しない方針（C案）|
|---|---|---|
| 初期構築コスト | 中央基盤構築工数 3-6 ヶ月 × 2-3 名 = **~$100K** | 各アプリで独立構築 × 10 = **~$300-500K** |
| 月額運用コスト | **~$1,143-1,253**（アプリ数増加でもあまり増えない）| **~$5,000-8,000**（アプリ数で線形増加）|
| Phase 1 末（10 アプリ）年間コスト | **~$14K/年** | **~$60-96K/年** |
| Phase 2 末（20 アプリ）年間コスト | **~$24K/年** | **~$120-192K/年** |

💡 **想定シナリオ**:

| ご判断 | 傾向 |
|---|---|
| 3-5 アプリ以下 | コスト差は小さく、C案 の初期投資少なめが有利になる場合も |
| 10 アプリ | **集約が明らかに安い**（10 倍以上の差）|
| 20+ アプリ | **集約が圧倒的に安い**（15 倍以上）|

⚠ **この観点の判断への影響**: **アプリ数 10+ なら集約有利、5 以下なら要議論**

### スライド 4-3: 総合比較 C. 運用（人・組織）

**タイトル**: 総合比較 C - 運用の観点

🎯 **このスライドで何を確認したいか**: **運用（人・組織）**の観点で 集約 / 個別 を比較

📝 **リード文**:
> 中央組織の要否・各アプリ側の負担・作業重複・共通マスタ管理の 5 点で比較します。特に**共通マスタ（顧客・組織・商品）の一元管理**が集約の大きな強みです。

📊 **表 4-3**: 運用（人・組織）の比較

| 観点 | 集約する方針（A案 / B案）| 集約しない方針（C案）|
|---|---|---|
| 中央組織 | 必要（2-5+ 名）| 不要 |
| 各アプリ側の負担 | データ加工人材必要（A案）or データ供給のみ（B案）| 各アプリで自足（結局はデータ人材必要）|
| スキル集約 vs 分散 | ⭕ 集約・標準化 | △ 分散、アプリ間でスキル差 |
| ダッシュボードの重複作業 | ⭕ 1 回作成、全社共有 | ❌ 各アプリで別々に作成 |
| 共通マスタ（顧客・組織・商品）の管理 | ⭕ 一元管理 | ❌ 各アプリで重複管理、整合性問題 |

💡 **想定シナリオ**:

| ご判断 | 傾向 |
|---|---|
| 中央組織を立ち上げる余力がある | 集約可能、A案 / B案 の選択へ |
| 中央組織の余力なし + 各アプリ人材あり | C案 or 集約自体を後回し |
| 共通マスタの重複管理が既に問題 | 集約が強く推奨 |
| ダッシュボード作業の重複を許容 | C案 でも問題なし |

⚠ **この観点の判断への影響**: **中央組織確保できるなら集約、できないなら C案**

**タイトル**: 総合比較 D - リスクの観点

🎯 **このスライドで何を確認したいか**: **リスク・その他**の観点で 集約 / 個別 を比較

📝 **リード文**:
> 障害影響範囲・単一障害点・新規アプリ追加・データガバナンス・ライセンスコストの 5 点で比較します。**リスクの構造が両案で真逆**である点にご注意ください。

📊 **表 4-4**: リスク・その他 の比較

| 観点 | 集約する方針（A案 / B案）| 集約しない方針（C案）|
|---|---|---|
| 障害影響範囲 | 中央障害で全社影響（Blast radius 大）| 1 アプリに閉じる（Blast radius 小）|
| 中央組織の単一障害点化 | △ あり | ⭕ なし |

| データガバナンス（GDPR/APPI 対応） | ⭕ 一元管理で対応しやすい | △ 各アプリで個別対応、抜け漏れリスク |
| ライセンスコスト（ダッシュボード画面等）| 1 セット | アプリ数分（10 セット）|

💡 **想定シナリオ**:

| ご判断 | 傾向 |

### スライド 4-7: 【補足】Catalog 登録 と ETL 変換の分離

**タイトル**: 補足 - 「Glue Catalog 登録」と「Glue ETL 変換」は別物

🎯 **このスライドで何を確認したいか**: よくある疑問「Athena 使途未定なのに ETL を書くのか?」への回答

📝 **リード文**:
> 前スライドの標準 3「Glue Catalog にテーブル登録」について、**「Athena での使い方（クエリ内容、ダッシュ設計）がまだ決まっていない中で ETL を書くのか?」というご質問への回答**を明確にしておきます。

📊 **表 4-7.1**: Catalog 登録 と ETL 変換 の位置付け

| 概念 | 内容 | 今やるか? | Athena 使途との関係 |
|---|---|:---:|---|
| **Glue Catalog に登録**（メタデータ）| テーブル定義（列名・型・場所）を記録するだけ、データは動かない | ⭕ **今やる**（数時間の DDL 登録） | **依存しない**（登録だけで発見可能に）|
| **Glue ETL で変換**（変換ロジック）| PII マスキング、集計、結合等の実際の加工処理 | ❌ **後で良い**（用途が見えてから）| **依存する**（用途駆動）|

📊 **表 4-7.2**: 「今やる」/「後で良い」の整理

| 分類 | 具体的な作業 | 実施タイミング |
|---|---|---|
| **今やる** | S3 バケット命名・レイアウト、Parquet 形式（Firehose / DMS の設定でできる、Glue ETL 不要）、パーティション、Glue Catalog の raw 層テーブル登録、機密度分類 | Phase 1 開始時 |
| **後で良い** | Glue ETL ジョブ（curated / analytics 層の集計・変換）、Athena Workgroup 設計、用途別ダッシュボード | Athena 使途決定後 |

💡 **結論**:
- **Athena 使途未定でも Glue Catalog 登録は今やる意味があります**（メタデータのみ、データは変換されない、後から追加してもデータには影響しない）
- **Glue ETL は用途駆動で後から書けば良い**（Data Lake 型設計の利点）
- Parquet 変換は **Firehose / DMS / Aurora Export の設定**でできる（Glue ETL 不要）
- 詳細は補足資料 [§2.5](data-collection-standards-for-future-aggregation.md) 参照

⚠ **想定される質問への回答**: 「用途未定なのに ETL を書くの?」→ **書かない**。Catalog 登録（メタデータ）のみ、ETL は用途決定後に追加。

### スライド 5: A案 分散型 の全体像

**タイトル**: A案 分散型 - 各アプリ側がデータを持ったまま、中央から見えるようにする

🎯 **このスライドで何を確認したいか**: A案（分散型）の思想の理解

📝 **リード文**:
> A案は、**中央集約するが データの物理的な移動はしない**構成です。各アプリ側は自分のデータを持ったまま、中央から「見える化」の許可を付けて、中央の分析画面から横断的に見えるようにします。**業務領域を持つ人が自分のデータを責任持って管理する**という考え方（業界では「業務領域別所有」と呼ばれます）が背景にあります。

📊 **図解**: A案 分散型

```
[各アプリ側 (既存アカウント)]              [中央 (新規追加 +1)]
┌─────────────────────────┐        ┌─────────────────────────┐
│ アプリ 1                     │        │ 中央データ管理             │
│  ├ 業務データベース           │        │  ├ アクセス許可管理         │
│  ├ データ変換処理             │◀──────│  ├ カタログ (データ目録)   │
│  └ S3 データ保管 (生・整形済) │  「見える│  └ 暗号鍵管理              │
├─────────────────────────┤   化」  │                            │
│ アプリ 2                     │        │ 中央分析                  │
│  ├ 業務データベース           │◀──────│  ├ 分析クエリサービス      │
│  ├ データ変換処理             │        │  ├ ダッシュボード画面      │
│  └ S3 データ保管             │        │  └ 派生データ保管         │
├─────────────────────────┤        └─────────────────────────┘
│ アプリ N ... (同じ構造)        │
└─────────────────────────┘
       ↑                                        ↑
   [各アプリ側チームが加工]              [中央分析チームは 2-3 名で成立]
```

**A案 の特徴**:

| 良いところ | 難しいところ |
|---|---|
| ⭕ **横断分析が可能** | △ 各アプリ側にデータ加工能力が必要 |
| ⭕ 中央チームは少人数（2-3 名）で成立 | △ 各アプリ側と中央の連携設定がやや複雑 |
| ⭕ データが動かないので中央障害でも喪失なし | |
| ⭕ 新規アプリ追加時、各アプリ側が自分で動ける | |
| ⭕ 業界標準（Netflix / Spotify 等が採用）| |

**必要な体制**:
- 中央分析チーム: **2-3 名**（少人数で OK）
- 各アプリ側: 各チームに **データ加工できる人が 1-2 名**

### スライド 6: B案 中央集約型 の全体像

**タイトル**: B案 中央集約型 - 全データを中央に集めて、中央で加工・分析する

🎯 **このスライドで何を確認したいか**: B案（中央集約型）の思想の理解

📝 **リード文**:
> B案は、**各アプリのデータを全て中央に書き写し**、中央でデータ加工・分析まで完結する構成です。中央にデータレイク（データを溜める大きな倉庫）を作り、そこに全部のデータを集めます。**中央の強力なチームが全社データを一元管理**するという伝統的な考え方です。銀行系や大企業の従来型データウェアハウスの延長線にあります。

📊 **図解**: B案 中央集約型

```
[各アプリ側 (既存アカウント)]              [中央 (新規追加 +1)]
┌─────────────────────────┐        ┌─────────────────────────┐
│ アプリ 1                     │        │ 中央データレイク           │
│  └ 業務データベース           │  複製→ │  ├ S3 生データ保管         │
├─────────────────────────┤  (書き│  ├ S3 整形済データ保管     │
│ アプリ 2                     │  写す)  │  ├ S3 集計データ保管       │
│  └ 業務データベース           │        │  ├ データ変換処理 (中央)   │
├─────────────────────────┤        │  ├ カタログ・許可管理      │
│ アプリ N ... (同じ構造)        │        │  └ 分析クエリ・画面        │
└─────────────────────────┘        └─────────────────────────┘
       ↑                                        ↑
   [各アプリ側は「データを渡すだけ」]     [中央分析チームは 5+ 名必要]
```

**B案 の特徴**:

| 良いところ | 難しいところ |
|---|---|
| ⭕ **横断分析が可能** | ❌ 中央チームの負担が非常に大きい |
| ⭕ 各アプリ側の作業が最小（データを渡すだけ）| ❌ 中央の障害で全社の分析が止まる |
| ⭕ 中央で自由に横断集計できる | ❌ 案件数が増えると中央が詰まる |
| ⭕ 案件数が少なければ運用が単純 | ❌ データを二重保管する（ストレージ費用増）|

**必要な体制**:
- 中央分析チーム: **5 名以上**（全社データを扱うため強力なチームが必要）
- 各アプリ側: データを渡す担当がいれば OK（データ加工スキル不要）

## §2 スライド作成時の注意点

### 2.1 用語の日本語化ルール

| 技術用語 | スライドでの表記 |
|---|---|
| Producer | 各アプリ側 / データ提供元 |
| Central / Consumer | 中央（分析・カタログ）|
| Federated Data Mesh | 分散型 / 業務領域別所有 |
| Central Data Lake | 中央集約型 / 中央データレイク |
| Fully Distributed | 個別型 / 独立運用 / サイロ型 |
| ETL | データ加工処理 |
| S3 | S3（クラウドストレージ）|
| Lake Formation | データ管理サービス / 許可管理 |
| Glue | データ変換サービス |
| Athena | 分析クエリサービス |
| QuickSight | 分析画面（ダッシュボード）サービス |
| SPICE | 高速キャッシュ |
| KMS | 暗号鍵管理サービス |
| CloudTrail | 監査ログサービス |
| Domain Ownership | 業務領域別所有 |
| Blast radius | 障害影響範囲 |
| LTV | 顧客生涯価値 |
| MTTR | 障害復旧までの平均時間 |
| CPA | 顧客獲得単価 |
| ROI | 投資対効果 |

### 2.2 進め方のポイント

| 原則 | 内容 |
|---|---|
| **Part 1 と Part 2 を明確に区別** | 「集約するか」と「A案/B案 の選択」は別の判断 |
| **Part 1 は経営・事業視点** | 技術ではなくビジネス価値で判断 |
| **Part 2 は体制・実装視点** | 技術的に成立するかを体制で確認 |
| **C案（集約しない）も等しく選択肢** | 「集約が正解」の前提を持たず、フェアに提示 |
| **その場で暫定判断まで進める** | スライド 4 で Part 1 暫定判断、スライド 9 で Part 2 暫定判断 |
| **叩き台を必ず提示** | 想定回答例を示すことで議論の収束を早める |

### 2.3 想定される難しいシナリオ

| シナリオ | 対応 |
|---|---|
| ビジネス価値の金額換算ができない | 定性的な期待効果のみで判断、経営持ち帰り |
| 集約価値ありだが中央 2 名も確保できない | Phase 1 縮小（3 アプリ）or C案 に戻す |
| 監査アカウント計画が全く見えない | Option C の追加コストと ROI 説明、持ち帰り |
| 各アプリ側スキルが大半不足 + 中央も 2 名 | Part 1 の再検討（集約自体の見直し）|
| 案件数が 3 個以下で確定 | C案 が最有力、A/B案 は将来案件数増を見越して概念設計のみ |
