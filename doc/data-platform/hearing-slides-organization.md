## Part 1: 集約するかしないか

### スライド 前段-1: データプラットフォームとは何か

**タイトル**: データプラットフォームとは何か

🎯 **このスライドで何を確認したいか**: データプラットフォームの基本概念の共通認識化

📝 **リード文**:
> **データプラットフォームとは、複数のシステムに散在しているデータを集めて統合し、分析できる状態にする仕組み**です。目的は「経営判断・業務改善・顧客体験向上のためのデータ活用」です。今、弊社では **10 個の SaaS 製品**があり、それぞれ独自にデータを持っていますが、**「全社を見渡した分析」ができない**状態です。

📊 **図解**: データプラットフォームの概念

```
[分散状態(現在)]                             [統合状態(データプラットフォーム後)]

┌─────┐  ┌─────┐                        ┌───────────────┐
│ SaaS 1│  │ SaaS 2│                        │ 経営: 全社 KPI │
│ データ │  │ データ │                        │ CS:  横断把握   │
└─────┘  └─────┘                        │ マーケ: ファネル │
┌─────┐  ┌─────┐  ────統合───→│ 会計: 売上合算 │
│ SaaS 3│  │ SaaS N│                        │ 開発: 品質統合 │
│ データ │  │ データ │                        └───────────────┘
└─────┘  └─────┘                                    ↑
    ↑                                        統合・分析
[各 SaaS で個別分析]                          [データプラットフォーム]
```

📊 **表 前段-1**: データプラットフォームの構成要素

| 構成要素 | 役割 |
|---|---|
| **データ収集** | 各 SaaS の業務データを集める（DB → S3 等）|
| **データ保管** | 集めたデータを長期保管（クラウドストレージ）|
| **データ加工** | 分析用に整形・集計・PII マスキング |
| **カタログ**（データ目録） | どこに何があるかの一元管理 |
| **分析クエリ** | SQL 等でデータを問い合わせる仕組み |
| **可視化** | ダッシュボード・レポート |

💡 **弊社での具体例**:

| 現在（個別対応） | データプラットフォーム後 |
|---|---|
| 経費精算 SaaS の売上、勤怠 SaaS の売上、CRM SaaS の売上 → **バラバラで見ている** | **1 つの画面で全社売上の合算・分解が見える** |
| CS が 1 顧客の複数 SaaS 状況を把握 → **各 SaaS の管理画面を開いて手動集計** | **顧客ダッシュボードで全 SaaS 状況を一目で把握** |
| 各 SaaS でエラーログ確認 → **共通不具合の発見が遅れる** | **全 SaaS 横断で品質メトリクスを把握、共通不具合を早期発見** |

⚠ **次のスライド**: では「データプラットフォームを構築する / しない」でどんなメリット・デメリットがあるかを比較します

### スライド 前段-2: 構築するメリット vs 個別対応のメリット

**タイトル**: データプラットフォームを構築する / 構築しないで個別対応する のメリット比較

🎯 **このスライドで何を確認したいか**: **両方針のメリット・デメリットを対等に理解**していただく

📝 **リード文**:
> データプラットフォームは**必ず構築すべきものではありません**。**構築する場合と個別で対応する場合**それぞれのメリット・デメリットを踏まえて、弊社に適した方針を選ぶ必要があります。以下、両案のメリットを対等に比較します。

📊 **表 前段-2.1**: 構築する / しない のメリット・デメリット対比

| 観点 | データプラットフォームを構築する | 構築しない（個別対応する）|
|---|---|---|
| **できるようになること** | | |
| 全社横断分析 | ⭕ 全社視点で経営判断可能 | ❌ できない |
| 共通マスタ管理 | ⭕ 一元管理、整合性担保 | ❌ 各アプリで重複管理、整合性リスク |
| 新規分析ニーズ | ⭕ 1 回導入で全社適用 | ❌ 各アプリで個別対応、重複作業 |
| データガバナンス | ⭕ 一元的な PII・監査対応 | △ 各アプリで対応、抜け漏れリスク |
| **失うこと・課題** | | |
| 中央組織の必要性 | ❌ 新規 2-5 名の中央組織が必要 | ⭕ 現状の体制のまま |
| 初期投資 | ❌ 中央基盤構築で 3-6 ヶ月 | ⭕ ゼロ |
| アプリ独立性 | △ 中央依存が発生 | ⭕ 完全独立 |
| 障害影響範囲 | ❌ 中央障害で全社影響 | ⭕ 1 アプリに閉じる |

📊 **表 前段-2.2**: それぞれのメリットが効くケース

| 構築する方針が正解のケース | 個別対応する方針が正解のケース |
|---|---|
| 全社横断で経営判断したい | 各 SaaS が独立していて横断不要 |
| 顧客の全 SaaS 状況を統合したい | 各アプリ内での分析で十分 |
| 共通マスタ管理の課題が顕在化 | マスタ重複は許容可（データ量少）|
| データ量・アプリ数が増える見込み | 現状の 3-5 アプリで凍結 |
| データガバナンス強化が必要 | 現状の対応で足りる |

💡 **お伝えしたいこと**:
- **業界では「構築する」方針が主流**ですが、**弊社の状況次第で「個別対応」の方が良いケースも十分あります**
- **無理に構築しない選択**も立派な戦略です（構築コストが実効価値を上回れば逆効果）
- 大切なのは「**どちらのメリットが弊社にとって大きいか**」を見極めること

⚠ **次のスライド**: どちらのメリットが弊社にとって大きいか判断するための **判断条件** を整理します

### スライド 2: 2 つの方針 - 集約する / しない

**タイトル**: まずは「そもそも集約するか」から

🎯 **このスライドで何を確認したいか**: 「集約 / 個別」は独立した根本判断であることの共有

📝 **リード文**:
> データ活用基盤には **2 つの根本的な方針**があります。**中央に集約する方針**（各アプリのデータを横断で見られるようにする）と、**集約しない方針**（各アプリで独立してデータ活用する）です。どちらが良いかは、**ビジネス目的・期待価値・コスト・運用**の 4 観点で総合的に判断します。

📊 **表**: 2 方針の一言比較

| 観点 | **集約する方針**（A案 or B案）| **集約しない方針**（C案 個別型）|
|---|---|---|
| 全社横断で見える | ⭕ 可能 | ❌ 事実上不可 |
| 各アプリの独立性 | △ 中央依存あり | ⭕ 完全独立 |
| データを持つ場所 | 1 箇所（分散 or 中央）| 各アプリ個別 |
| 中央組織 | 必要（2-5 名以上）| 不要 |
| 初期投資 | 中央基盤構築（工数・アカウント設計）| 各アプリで独立に構築 |
| 月額運用コスト（10 アプリ想定）| ~$1,143-1,253 | ~$5,000-8,000（アプリ数で線形増加）|
| 障害影響範囲 | 中央障害で全社影響 | 1 アプリに閉じる |
| **業界通称** | 「データレイク」「データメッシュ」| 「独立運用」「サイロ型」 |

💡 **お伝えしたいこと**:

🎯 **このスライドで何を確認したいか**: C案 採用時の**将来の統合パス**の展望を共有

📝 **リード文**:
> C案 個別型 を選んだ場合でも、**各アプリ側で最低限の標準を守ることで、将来的にビジネス環境が変わって A案 / B案 に移行する必要が出てきた際の移行工数を 3-4 倍短縮できます**。追加コストはほぼゼロで、各アプリ内でも品質が上がります。詳細は補足資料をご参照ください。

📊 **表 4-7**: 将来の統合可能性を残すための 5 つの標準

| # | 標準 | 追加コスト | 将来メリット |
|---|---|---|---|
| 1 | 分析用データは S3 に保管、**Parquet + Snappy 圧縮** | ゼロ（ストレージ 75% 削減）| 将来の中央参照が容易 |
| 2 | **`tenant_id / year / month / day` のパーティション** | ゼロ（設計時に決めるだけ）| 中央からのアクセス制御・クエリコスト削減 |
| 3 | **AWS Glue Data Catalog にテーブル登録**（メタデータのみ）| ゼロ（無料枠内、数時間）| 将来の中央 Federation で機械的に連携 |
| 4 | 列名・型の共通化（`tenant_id`, `event_date` 等）| ゼロ（設計時に決めるだけ）| 統合時の変換工数ゼロ |
| 5 | **機密度分類**（Public/Internal/Confidential/Restricted）+ PII: Yes/No | ゼロ（付与するだけ）| 将来のガバナンス統合が容易 |

💡 **移行工数の比較**:

| 移行シナリオ | 標準を守っている場合 | 標準を守っていない場合 |
|---|---|---|
| C → A案 移行 | **3-6 ヶ月** | 12-24 ヶ月 |
| C → B案 移行 | **6-12 ヶ月** | 18-36 ヶ月 |
| 10 アプリ移行コスト | **$300-500K** | $1-3M（1-3 億円）|

⚠ **推奨**: C案 採用時は、上記 5 標準を各アプリ側の実装ガイドラインとして周知（詳細は [補足資料](data-collection-standards-for-future-aggregation.md)）

### スライド 4-8: 【補足】Catalog 登録 と ETL 変換の分離

**タイトル**: 補足 - 「Glue Catalog 登録」と「Glue ETL 変換」は別物

🎯 **このスライドで何を確認したいか**: よくある疑問「Athena 使途未定なのに ETL を書くのか?」への回答

📝 **リード文**:
> 前スライドの標準 3「Glue Catalog にテーブル登録」について、**「Athena での使い方（クエリ内容、ダッシュ設計）がまだ決まっていない中で ETL を書くのか?」というご質問への回答**を明確にしておきます。

📊 **表 4-8.1**: Catalog 登録 と ETL 変換 の位置付け

| 概念 | 内容 | 今やるか? | Athena 使途との関係 |
|---|---|:---:|---|
| **Glue Catalog に登録**（メタデータ）| テーブル定義（列名・型・場所）を記録するだけ、データは動かない | ⭕ **今やる**（数時間の DDL 登録） | **依存しない**（登録だけで発見可能に）|
| **Glue ETL で変換**（変換ロジック）| PII マスキング、集計、結合等の実際の加工処理 | ❌ **後で良い**（用途が見えてから）| **依存する**（用途駆動）|

📊 **表 4-8.2**: 「今やる」/「後で良い」の整理

| 分類 | 具体的な作業 | 実施タイミング |
|---|---|---|
| **今やる** | S3 バケット命名・レイアウト、Parquet 形式（Firehose / DMS の設定でできる、Glue ETL 不要）、パーティション、Glue Catalog の raw 層テーブル登録、機密度分類 | Phase 1 開始時 |
| **後で良い** | Glue ETL ジョブ（curated / analytics 層の集計・変換）、Athena Workgroup 設計、用途別ダッシュボード | Athena 使途決定後 |

💡 **結論**:
- **Athena 使途未定でも Glue Catalog 登録は今やる意味があります**（メタデータのみ、データは変換されない、後から追加してもデータには影響しない）
- **Glue ETL は用途駆動で後から書けば良い**（Data Lake 型設計の利点）
- Parquet 変換は **Firehose / DMS / Aurora Export の設定**でできる（Glue ETL 不要）
- 詳細は補足資料 [§2.5](data-collection-standards-for-future-aggregation.md) 参照

⚠ **想定される質問への回答**: 「用途未定なのに ETL を書くの?」→ **書かない**。Catalog 登録（メタデータ）のみ、ETL は用途決定後に追加。


## Part 2: 集約する場合の A案 / B案 選択

### スライド Part 2 前段: 処理はどこでやるか?

**タイトル**: 処理はどこでやるか? - 集約する場合の 2 つの分け方

🎯 **このスライドで何を確認したいか**: 「集約する」と決めた場合の **2 つのパターン** の顧客向けフレーミングの共通認識化

📝 **リード文**:
> 集約する方針が決まった場合、次に **「データの処理をどこでやるか」**で 2 つのパターンに分かれます。
> - **Pattern X**: 各アプリで処理を完結し、データプラットフォームは「参照 + AI」用途のみ（= A案 分散型）
> - **Pattern Y**: データをプラットフォームに集約してから処理を行う（= B案 中央集約型）
>
> どちらも**横断分析はできます**が、**処理の分担**が違います。

📊 **表 Part2 前段.1**: 作業別の分担マトリクス

| 作業段階 | Pattern X（各アプリ処理完結）= A案 分散型 | Pattern Y（プラットフォーム集中処理）= B案 中央集約型 |
|---|---|---|
| ① データ収集（取込）| 各アプリ | 各アプリ（→ プラットフォームへ複製）|
| ② クレンジング（整形・PII マスキング）| **各アプリ** | **プラットフォーム** |
| ③ 加工・集計（ドメイン別集計）| **各アプリ** | **プラットフォーム** |
| ④ 保管（S3 データ倉庫）| 各アプリの S3（参照許可経由）| プラットフォームの S3 |
| ⑤ 全社横断集計 | **プラットフォーム** | プラットフォーム |
| ⑥ 参照・可視化（ダッシュボード）| プラットフォーム | プラットフォーム |
| ⑦ AI / ML | プラットフォーム | プラットフォーム |

**→ ②③ の主体が違う**。それが Pattern X / Y の本質的な違いです。

📊 **表 Part2 前段.2**: それぞれのメリット・課題

| 観点 | Pattern X（各アプリ処理）| Pattern Y（プラットフォーム処理）|
|---|---|---|
| **メリット** | | |
| 各アプリの自律性 | ⭕ 高い（業務領域別所有）| △ 中央依存 |
| 中央組織の負担 | 小（2-3 名で成立）| 大（5+ 名必要）|
| 障害影響範囲 | 各アプリ単位に閉じる | 中央障害で全社影響 |
| データの物理移動 | なし（各アプリに残る）| あり（中央に複製、二重保管）|
| **課題** | | |
| 各アプリのスキル要求 | データ加工経験が必要 | 不要（データを渡すだけ）|
| 処理の標準化 | △ アプリ間でばらつく可能性 | ⭕ 統一しやすい |
| 案件数増加時のスケール | ⭕ アプリ側に負荷分散 | ❌ 中央がボトルネック化 |
| 新規アプリ追加時 | 中央連携設定（1-2 週間）| データ供給設定（数日）|

📊 **表 Part2 前段.3**: 顧客に「どちらが良いか」を示す判断基準

| ご状況 | 推奨パターン | 理由 |
|---|---|---|
| 各アプリチームがデータ処理可能 + 中央小規模 | **Pattern X（A案 分散型）** | 各アプリの自律性を保ちつつ、中央は少人数で運用 |
| 各アプリチームは薄い + 中央厚い（5+ 名）| **Pattern Y（B案 中央集約型）** | 中央で全部やることで各アプリの負担ゼロ |
| どちらも中間 | 経営判断 | Trade-off を経営が引き取る |
| 案件数が 10+ で急拡大予定 | **Pattern X 有利** | 中央ボトルネックを避けたい |
| 案件数が 3-5 で安定 | **Pattern Y でも成立** | 中央で捌ける規模 |

💡 **顧客向けのわかりやすい説明例**:

> 「集約すると決めた場合、次は **『データの前処理をどこでやるか』**を決めます。
>
> - **Pattern X**（A案 分散型）は、各アプリで責任持って処理していただき、プラットフォームは **参照・分析・AI 用途に特化** します。**各アプリの独立性を尊重**する構成です。

> どちらが御社に適しているかは、**体制（中央チームと各アプリの規模・スキル）**で決まります。次のスライドから、各案の詳細と、体制ヒアリングを進めさせてください。」

### スライド 5: A案 分散型 の全体像

**タイトル**: A案 分散型 - 各アプリ側がデータを持ったまま、中央から見えるようにする

🎯 **このスライドで何を確認したいか**: A案（分散型）の思想の理解

📝 **リード文**:
> A案は、**中央集約するが データの物理的な移動はしない**構成です。各アプリ側は自分のデータを持ったまま、中央から「見える化」の許可を付けて、中央の分析画面から横断的に見えるようにします。**業務領域を持つ人が自分のデータを責任持って管理する**という考え方（業界では「業務領域別所有」と呼ばれます）が背景にあります。

📊 **図解**: A案 分散型

```
[各アプリ側 (既存アカウント)]              [中央 (新規追加 +1)]
┌─────────────────────────┐        ┌─────────────────────────┐
│ アプリ 1                     │        │ 中央データ管理             │
│  ├ 業務データベース           │        │  ├ アクセス許可管理         │
│  ├ データ変換処理             │◀──────│  ├ カタログ (データ目録)   │
│  └ S3 データ保管 (生・整形済) │  「見える│  └ 暗号鍵管理              │
├─────────────────────────┤   化」  │                            │
│ アプリ 2                     │        │ 中央分析                  │
│  ├ 業務データベース           │◀──────│  ├ 分析クエリサービス      │
│  ├ データ変換処理             │        │  ├ ダッシュボード画面      │
│  └ S3 データ保管             │        │  └ 派生データ保管         │
├─────────────────────────┤        └─────────────────────────┘
│ アプリ N ... (同じ構造)        │
└─────────────────────────┘
       ↑                                        ↑
   [各アプリ側チームが加工]              [中央分析チームは 2-3 名で成立]
```

**A案 の特徴**:

| 良いところ | 難しいところ |
|---|---|
| ⭕ **横断分析が可能** | △ 各アプリ側にデータ加工能力が必要 |
| ⭕ 中央チームは少人数（2-3 名）で成立 | △ 各アプリ側と中央の連携設定がやや複雑 |
| ⭕ データが動かないので中央障害でも喪失なし | |
| ⭕ 新規アプリ追加時、各アプリ側が自分で動ける | |
| ⭕ 業界標準（Netflix / Spotify 等が採用）| |

**必要な体制**:
- 中央分析チーム: **2-3 名**（少人数で OK）
- 各アプリ側: 各チームに **データ加工できる人が 1-2 名**

### スライド 6: B案 中央集約型 の全体像

**タイトル**: B案 中央集約型 - 全データを中央に集めて、中央で加工・分析する

🎯 **このスライドで何を確認したいか**: B案（中央集約型）の思想の理解

📝 **リード文**:
> B案は、**各アプリのデータを全て中央に書き写し**、中央でデータ加工・分析まで完結する構成です。中央にデータレイク（データを溜める大きな倉庫）を作り、そこに全部のデータを集めます。**中央の強力なチームが全社データを一元管理**するという伝統的な考え方です。銀行系や大企業の従来型データウェアハウスの延長線にあります。

📊 **図解**: B案 中央集約型

```
[各アプリ側 (既存アカウント)]              [中央 (新規追加 +1)]
┌─────────────────────────┐        ┌─────────────────────────┐
│ アプリ 1                     │        │ 中央データレイク           │
│  └ 業務データベース           │  複製→ │  ├ S3 生データ保管         │
├─────────────────────────┤  (書き│  ├ S3 整形済データ保管     │
│ アプリ 2                     │  写す)  │  ├ S3 集計データ保管       │
│  └ 業務データベース           │        │  ├ データ変換処理 (中央)   │
├─────────────────────────┤        │  ├ カタログ・許可管理      │
│ アプリ N ... (同じ構造)        │        │  └ 分析クエリ・画面        │
└─────────────────────────┘        └─────────────────────────┘
       ↑                                        ↑
   [各アプリ側は「データを渡すだけ」]     [中央分析チームは 5+ 名必要]
```

**B案 の特徴**:

| 良いところ | 難しいところ |
|---|---|
| ⭕ **横断分析が可能** | ❌ 中央チームの負担が非常に大きい |
| ⭕ 各アプリ側の作業が最小（データを渡すだけ）| ❌ 中央の障害で全社の分析が止まる |
| ⭕ 中央で自由に横断集計できる | ❌ 案件数が増えると中央が詰まる |
| ⭕ 案件数が少なければ運用が単純 | ❌ データを二重保管する（ストレージ費用増）|

**必要な体制**:
- 中央分析チーム: **5 名以上**（全社データを扱うため強力なチームが必要）
- 各アプリ側: データを渡す担当がいれば OK（データ加工スキル不要）

### 2.4 顧客への説明スクリプト（判断結果別のトーキング）

**位置付け**: スライド 4-6 の総合判定を踏まえて、判定結果を顧客に伝える際の推奨スクリプト。特に「横断要件がなく、C案 で確定」となった場合に、**「不要ですね」と突き放した印象にならず、戦略的な選択として伝える**ためのフレーミング。

#### 2.4.1 スマートに伝えるための 4 つの原則

| 原則 | ❌ 悪い例 | ⭕ 良い例 |
|---|---|---|
| **主語は顧客の状況** | 「うちの提案は不要ですね」 | 「御社の現状では〜が最適です」 |
| **選択肢として提示** | 「作らないほうがいいです」 | 「無理に構築しない選択も、立派な戦略です」|
| **将来へのオープン** | 「今後も要らないでしょう」 | 「将来必要になった時にスムーズ移行できる準備をご提案します」|
| **判断基準を透明に** | 「今は要らないと思います」 | 「以下の状況が発生したら再検討ください」 |

#### 2.4.2 パターン A: 横断要件が明らかに薄い場合（C案 で確定）

> 「本日 5 観点でお伺いした結果、**御社は現時点で全社横断分析のご要望が明確に浮かび上がりませんでした**。
>
> このような場合、**中央データプラットフォームの構築を急ぐ必要はない**と考えています。無理に構築しない選択も、業界的に**立派な戦略の一つ**です。理由は 3 つあります:
>
> 1. **投資対効果**: 横断価値が薄い中で年 $14K の投資は妥当性を欠く
> 2. **組織影響**: 新規 2-5 名の中央組織を設置しなくて済む
> 3. **柔軟性**: 各 SaaS 製品が独立して自由に改善できる状態を維持
>
> 一方で、将来ビジネス環境が変わった時に慌てないよう、**最低限の 5 標準**（S3 + Parquet + パーティション + Glue Catalog 登録 + 機密度分類）を各アプリで守っていただければ、将来の統合工数を **3-4 倍短縮**できます。追加コストほぼゼロです。
>
> 移行判断のトリガーもお示ししておきます: **経営陣から全社 KPI 要望 / 顧客からの横断要望 / SaaS 製品数 5+ / 業界規制の変化**。この 4 つのうちいずれかが発生した時に、再検討させてください。」

**話す時のコツ**:
- 「業界的に立派な戦略」で権威付け（顧客に恥をかかせない）
- **数字を具体的に**（年 $14K、2-5 名、5 標準、3-4 倍）で説得力を出す
- **明確な再検討トリガー**を提示（放置ではなく計画的な選択と示す）

#### 2.4.3 パターン B: 横断要件が微妙・混在の場合（持ち帰り推奨）

> 「本日の議論で、5 観点のうち **X 個は構築有利、Y 個は個別有利**、と傾向が混在しました。
>
> このような場合、**その場で結論を急がず、経営会議で持ち帰り判断**していただくのが良いと考えています。特に **『期待価値の金額換算』**（スライド 3 参照）が明確になると、判断がしやすくなります。
>
> 2 週間以内に判断されるまでの間、**個別対応の準備**を並行して進めておくことを推奨します。仮に将来『構築する』となっても、この準備は無駄になりませんし、『個別対応』となってもそのまま活用できます。
>
> 判断まで期間を要する場合、その間の各アプリでのデータ蓄積が **将来の統合可能性を左右**します。5 標準を守っていただければ、どちらに転んでも柔軟に対応できます。」

**話す時のコツ**:
- 経営判断へのボールパスをスマートに（弊社が引き取れない領域を明示）
- 「並行準備」で **ムダにならない**ことを保証
- 判断期間中も **何かしら価値を蓄積**できることを示す

#### 2.4.4 パターン C: 横断要件が明確にある場合（A案/B案 選択へ）

> 「本日の議論で、**5 観点のうち X 個が構築有利**という明確な傾向が確認できました。特に **『経営 KPI 一元化』『CS 横断把握』**へのご要望が強く、これは中央データプラットフォームでしか実現できません。
>
> 集約する方針で進めさせていただきます。Part 2 では、**A案 分散型 / B案 中央集約型 のどちらが体制的にフィットするか**を確認していきます。」

**話す時のコツ**:
- **顧客自身の言葉**（横断把握、経営 KPI 等）を引用して確認感を出す
- 次のステップを **すぐ提示**して勢いを維持

#### 2.4.5 C案 を「戦略的な選択」として位置づけるフレーミング

C案 が「消極的な選択」ではなく「**戦略的な選択**」として位置づけるための言い回し集:

| フレーミング | 使い方 |
|---|---|
| **「シンプルさこそ強み」** | 「複雑な中央基盤を持たず、各 SaaS が自律的に動く体制は、意思決定が早く柔軟性が高い」 |
| **「投資は必要な時に集中」** | 「横断ニーズが顕在化した時に、確信を持って集中投資できるよう、今は資源を温存」 |
| **「業界事例の紹介」** | 「実は、大規模 SaaS 企業でも敢えて集約しない選択をしている事例が多数あります」 |
| **「弊社の推奨」** | 「弊社としても、御社の状況では C案 が最も投資効率が高いとお勧めします」 |

#### 2.4.6 想定される顧客からの反応と対応

| 顧客の反応 | 対応 |
|---|---|
| 「本当に作らなくていいの?」（不安）| 移行トリガー 4 つを再度示し、「変化を検知したら即座に再開できる」体制を強調 |
| 「他社は作っているのでは?」（同調圧力）| 「事例は多いですが、御社の規模・状況では最適解が異なります。無理に他社に合わせる必要はありません」 |
| 「投資してもいいけど…」（曖昧）| 「価値が薄い時に投資すると ROI が悪く、社内で評価されにくくなります。明確な価値が見えてから投資する方が、経営的にも説明しやすくなります」 |
| 「何もしないのはもったいない」（アクション欲求）| 「5 標準の実施を提案します。これは低コストの布石で、無駄になりません。将来必要になった時に効いてきます」 |
