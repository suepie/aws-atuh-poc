### §C-2.0.A 本標準のスタンス

> **AWS ネイティブを原則とし、機能カテゴリごとに「標準サービス」を明示する。SaaS は 3 年 TCO で 30% 以上のメリット or 他の代替不可な機能要件がある場合のみ ADR で例外採用を許容。基本方針 4 軸（安全 / 網羅 / 効率 / コスト）の各軸での評価を選定時に必須化する。**

## §C-2.1 評価基準

> **このサブセクションで定めること**: サービス選定時に必ず評価する 6 軸。
> **主な判断軸**: 評価の網羅性 / 判断のぶれない明文化
> **§C-2 全体との関係**: 選定意思決定の共通言語

### ベースライン

| # | 評価軸 | 重み | 説明 |
|---|---|---|---|
| 1 | 絶対安全（基本方針）| 高 | 暗号化・アクセス制御・監査要件の充足 |
| 2 | どんなユースケースでも（基本方針）| 中 | データ区分・アクセスパターンの網羅 |
| 3 | 効率（基本方針）| 中 | 開発・運用フリクションレス |
| 4 | 運用負荷・コスト（基本方針）| 高 | マネージド度・TCO |
| 5 | 既存資産との統合 | 中 | 移行コスト・スキルセット |
| 6 | ベンダーロックイン | 低 | 将来移行可能性 |

## §C-2.2 主要論点と標準サービス

> **このサブセクションで定めること**: 機能カテゴリ別の標準サービスと、それを選ぶ理由。
> **主な判断軸**: §C-2.1 6 軸での評価結果
> **§C-2 全体との関係**: §FR-2〜4 の選定結果のサマリー

### ベースライン

| 機能カテゴリ | 標準サービス | 代替検討対象 | 選定の主な理由 |
|---|---|---|---|
| データレイク | S3 + Glue Catalog | (固定) | 唯一の選択肢 |
| DWH | Redshift（Serverless / プロビジョンド） | Snowflake（不採用） | ネイティブ・コスト |
| クエリエンジン | Athena | Redshift Spectrum / Trino on EMR | サーバレス・低コスト |
| ETL | Glue ETL | EMR / Lambda / Step Functions | Spark マネージド |
| ストリーム | Kinesis Data Streams / Firehose | MSK / EventBridge Pipes | サーバレス・コスト |
| Kafka 互換 | MSK / MSK Serverless | Confluent Cloud（不採用） | Kafka 互換が要件のときのみ |
| CDC | DMS / Zero-ETL | Striim 等 SaaS（不採用） | ネイティブ・無料 |
| BI | QuickSight | Tableau / Looker（不採用原則） | ネイティブ・統合 |
| 検索 | OpenSearch Service | Elastic Cloud（不採用） | ネイティブ・統合 |
| メタデータ管理 | Glue Data Catalog / Lake Formation | Atlan / Collibra（不採用） | ネイティブ・統合 |
| データ品質 | Glue Data Quality | Soda / Great Expectations | ネイティブ優先 |

## §C-2.3 SaaS 採用例外条件

> **このサブセクションで定めること**: SaaS 採用が許容される条件と、採用時の ADR 要件。
> **主な判断軸**: TCO / 機能要件の代替不可性 / ベンダーロックインリスク
> **§C-2 全体との関係**: 標準逸脱の唯一の正規ルート

### ベースライン

**SaaS 採用許容条件**（全て満たすこと）:
- 3 年 TCO で AWS ネイティブを **30% 以上**上回るメリット、または
- AWS ネイティブで代替不可な機能要件がある（具体的に明示）
- ベンダーロックインリスクの評価結果が許容範囲内
- 既存資産との整合性が確認済み

**ADR 必須項目**:
- 採用理由（TCO 計算 or 代替不可機能の明示）
- 代替案（AWS ネイティブ）との比較
- ベンダーロックインリスクと退避戦略
- 契約期間・退出条件
