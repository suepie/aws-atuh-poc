# 【補足資料】将来の統合に備えたデータ収集の標準

> **位置付け**: 「集約しない方針（C案 個別型）」を採用する場合でも、**将来的に集約が必要になった時に困らないよう**、各アプリで最低限守っておくべきデータ収集の標準を示す資料。
> **対応スライド**: [hearing-slides-organization.md](hearing-slides-organization.md) スライド 4 の補足として参照
> **主な想定利用者**: 各アプリ側のデータエンジニア / プロダクトマネージャ / 経営層
> **作成日**: 2026-07-02

---

## §0 この資料の背景

### 0.1 なぜこの資料が必要か

Part 1 の判断で「**集約しない（C案 個別型）**」を選んだとしても、**将来的にビジネス環境が変わって集約が必要になる**可能性は十分あります:

| 想定シナリオ | 集約が必要になる契機 |
|---|---|
| ① 事業成長で SaaS 製品数が増える | アプリ 3 個 → 10 個 に成長 |
| ② 経営陣が交代 / 事業戦略変更 | 「全社 KPI を横断で見たい」の要望が新たに発生 |
| ③ 大口顧客の要望 | 「複数 SaaS の横断請求書」等 |
| ④ 業界規制の変化 | GDPR/APPI 対応で一元管理が必要に |
| ⑤ 他社買収・グループ再編 | 新たなアプリ群との統合が必要 |
| ⑥ 競合対抗 | 「AI で解約予兆検知」等の要望 |

そのとき、**各アプリのデータ収集が バラバラだと統合が困難** になり、統合コストが数倍〜数十倍に跳ね上がります。

### 0.2 この資料の目的

- 集約しない場合でも **将来の統合可能性を残す** ための最低限の標準を示す
- 各アプリで **今から守っておくと得** な設計原則を提示
- **標準を守る追加コストはほぼゼロ**、むしろ各アプリ内でも品質が上がる

### 0.3 対象読者と使い方

| 読者 | 使い方 |
|---|---|
| 経営層 | §1 の全体像を確認、「集約しない」判断の後の展望を理解 |
| 各アプリ側データエンジニア | §2-§4 の技術標準を各アプリで実装 |
| プロダクトマネージャ | §5 のロードマップで将来移行の見通しを把握 |
| データ活用基盤側 | 各アプリへの推奨事項として提示、ヒアリングで確認 |

## §1 全体像: 「集約しない」を選んでも将来の道は残せる

### 1.1 3 つの将来シナリオ

集約しない方針を採用した後の 3 つの将来シナリオ:

```
現在: C案 個別型 で開始
    │
    ├─ シナリオ A: 5-10 年間 C案 継続
    │  → ビジネス環境が変わらなければ そのまま
    │
    ├─ シナリオ B: 数年後 A案（分散型）に移行
    │  → 中央集約基盤を構築、既存 S3 データを Federation で連携
    │  → 移行工数: 「標準を守っている」場合 3-6 ヶ月、守っていない場合 12-24 ヶ月
    │
    └─ シナリオ C: 数年後 B案（中央集約型）に移行
       → 中央にデータを集約、既存 S3 データを移動
       → 移行工数: 「標準を守っている」場合 6-12 ヶ月、守っていない場合 18-36 ヶ月
```

### 1.2 「標準を守る」vs「守らない」の移行コスト差

| 移行シナリオ | 標準を守っている場合 | 標準を守っていない場合 | 差分 |
|---|---|---|---|
| C → A案 移行 | 3-6 ヶ月 | 12-24 ヶ月 | **3-4 倍** |
| C → B案 移行 | 6-12 ヶ月 | 18-36 ヶ月 | **3 倍** |
| 追加工数（金額換算、1 アプリあたり）| $30-60K | $120-360K | **数倍〜10 倍** |

→ **10 アプリあれば $1-3M（1-3 億円）の差**になる可能性があります。**将来の統合可能性を残すコストはほぼゼロ**、むしろ各アプリの品質が上がるため実質プラスです。

### 1.3 大原則: 「クラウドの標準を素直に使う」

将来の統合可能性を残すコツは、**特殊な選択をしない、標準を素直に使う** ことです:

| 標準的な選択 | 特殊な選択（避けるべき）|
|---|---|
| ⭕ S3 でデータ保管 | ❌ 独自 NoSQL、独自バイナリ形式 |
| ⭕ Parquet ファイル形式 | ❌ 独自バイナリ、圧縮 CSV のみ |
| ⭕ 標準 SQL でクエリ | ❌ 独自クエリ言語、GUI 専用 |
| ⭕ AWS Glue Catalog でメタデータ管理 | ❌ 各アプリ独自の管理台帳 |
| ⭕ 業界標準の分類（機密度、PII 等）| ❌ 各アプリ独自の分類 |
| ⭕ IAM Role で認可制御 | ❌ API キー直書き |

## §2 データ保管の標準（S3 レイアウト・形式）

### 2.1 標準 1: S3 に保管する（独自 DB / 独自ストレージは避ける）

**推奨**:
- データはできる限り **S3 に保管** する（業務 DB からは日次・時間次で S3 に書き出す）
- 業務 DB（Aurora / DynamoDB）に留めず、分析用は S3 に分離

**理由**:
- S3 は AWS 全サービスから最も広く読める（Athena / Redshift / EMR / SageMaker / QuickSight 全対応）
- 将来、中央から読みに行くとき「S3 経由」が最も柔軟

**避けるべき**:
- 分析用データを RDS 内に留める → 将来の中央からの参照が困難
- 独自ストレージサービス（EBS 直接、独自バイナリ等）

### 2.2 標準 2: バケット命名とレイアウト

**推奨バケット命名**:
```
<会社略称>-<アプリ名>-<データ層>-<環境>
例:
  acme-expense-raw-prd       (生データ層)
  acme-expense-curated-prd   (整形済み層)
  acme-expense-analytics-prd (集計・分析用層)
```

**推奨レイアウト**（Medallion 3 層）:
```
s3://acme-expense-analytics-prd/
├── expenses/                   ← テーブル名
│   ├── tenant_id=T-001/       ← 顧客テナント別
│   │   ├── year=2026/
│   │   │   ├── month=07/
│   │   │   │   └── day=02/
│   │   │   │       └── data.parquet
│   │   │   └── ...
│   │   └── ...
│   ├── tenant_id=T-002/
│   └── ...
├── users/
└── ...
```

**理由**:
- 命名規則があると、将来の中央から「どのバケットに何があるか」を機械的に把握できる
- レイアウトが揃っていると、Federation 設定や移行スクリプトが機械的に書ける

### 2.3 標準 3: ファイル形式 = Parquet + Snappy 圧縮

**推奨**: 分析用データは **Parquet 形式** で保管、**Snappy 圧縮** を適用

**理由**:
- Parquet は業界標準の**列指向形式**、Athena / Redshift / Snowflake / Databricks 全対応
- Snappy 圧縮でストレージ 75% 削減
- スキーマ情報がファイルに埋め込まれるため、将来の再解析が容易

**避けるべき**:
- 独自バイナリ形式
- 圧縮なし CSV（サイズが 4 倍）
- Excel（xlsx）を長期保管形式にする

### 2.4 標準 4: パーティション設計

**推奨パーティション**（マルチテナント SaaS の場合）:
```
tenant_id=<顧客企業ID>/year=<YYYY>/month=<MM>/day=<DD>/
```

**理由**:
- `tenant_id` パーティションは**将来の中央からのアクセス制御**（Lake Formation Data Filter 等）で必須
- 日付パーティションは **クエリコストを 100 倍〜10,000 倍削減**
- 統合時にこの構造がバラバラだと変換工数が莫大

**避けるべき**:
- パーティションなし → 全走査でコスト爆発
- `tenant_id` を列に持たず、テナント別にファイル分割 → 中央からの結合が困難

## §2.5 【重要 FAQ】C案 では Glue Catalog まで用意せず、S3 のみでも大丈夫か?

### 2.5.0 前提: 「Glue Catalog に登録」と「Glue ETL で変換」は別物

**混同されがちな 2 つの概念**を明確に分けます:

| 概念 | 内容 | 実施タイミング | Athena 使途との関係 |
|---|---|---|---|
| **① Glue Catalog に登録**（メタデータ）| テーブル定義（列名・型・場所）を記録するだけ | **今すぐ実施可能**（数時間）| **依存しない**（登録だけで発見可能に）|
| **② Glue ETL でデータ変換**（実際の加工処理）| PII マスキング、集計、結合等の変換ロジック | **Athena での使い方が見えてから** | **依存する**（用途駆動）|

→ **「Athena 使途が決まっていないので Glue も後回し」というのは誤解**。**Glue Catalog 登録は用途と独立に「今」やる意味がある**、**Glue ETL は用途が見えてから作る**、が正解です。

### 2.5.1 結論

**技術的には可能だが、4 つのサブケースに分かれ、各アプリの状況で推奨が変わります**。

| サブケース | 実施内容 | Glue ETL | 各アプリの分析利用 | 移行工数（→ A/B案）|
|---|---|:---:|:---:|:---:|
| **C-1: S3 のみ**（Catalog なし）| データを溜めるだけ | なし | ❌ 事実上不可 | **18-36 ヶ月** |
| **C-2: S3 + raw層 Catalog**（**新・推奨下限**）| **raw のみ Catalog 登録、ETL なし** | **なし** | ⭕ 生データを直接クエリ可能 | **6-12 ヶ月** |
| **C-2+: S3 + raw + curated**（余力があれば）| 汎用クリーニング済 curated 層まで | 軽微（PII マスク等）| ⭕ curated 済データをクエリ | **3-6 ヶ月** |
| **C-3: フルセット**（Crawler + Data Quality + analytics）| S3 + Catalog + 自動検出 + 用途別集計 | 大（用途駆動）| ⭕ 本格運用 | **2-4 ヶ月** |

### 2.5.2 重要な事実 その 1: Athena を使うなら Glue Catalog は「事実上必須」

- **Amazon Athena は Glue Data Catalog を前提**に動作します
- `CREATE EXTERNAL TABLE` は裏側で Glue Catalog にエントリを作る仕様
- **各アプリで少しでも SQL 分析するなら Glue Catalog は自動的に必要**
- **QuickSight / SageMaker から S3 を見る場合も Athena 経由 = Glue Catalog 必要**

つまり「Glue Catalog を使わない = SQL 分析を一切しない」を意味します。

### 2.5.2b 重要な事実 その 2: Catalog 登録に Athena 使途は不要

- **Glue Catalog 登録は「変換」ではなく「目録作成」**です
- **データは変換されない、動かない、増えない、コストも増えない**
- 「こういう列を持つデータが S3 のここにあります」と記録するだけ
- **どんな Athena クエリでも使え、後で使途が決まった時に発見・活用できる**

```sql
-- これは変換ではなく、S3 に既にあるデータの「場所と形」を登録するだけ
CREATE EXTERNAL TABLE expense_app_raw.expenses (
    expense_id STRING,
    amount BIGINT,
    ...
)
LOCATION 's3://acme-expense-raw-prd/expenses/';
-- ↑ このコマンドはデータには一切触れません。メタデータだけ記録します
```

→ **「用途未定なので Glue Catalog は後回し」は誤解**。用途未定でも Catalog 登録には意味があります。

### 2.5.2c 重要な事実 その 3: Glue ETL は用途駆動で後から作れば良い

- **PII マスキング / tenant_id 強制 / 重複排除** 等の**汎用処理** → 用途未定でも書ける
- **業務別集計 / KPI 集計 / 結合テーブル** 等の**用途別処理** → 用途決定後に書く
- **後から追加しても既存の raw 層データには影響なし**（Glue ETL は raw を読んで別層を作るだけ）

つまり **「今 Glue ETL を書かなくても後から書ける」** が Data Lake 型設計の利点です。

### 2.5.3 サブケース別の追加コスト・工数

| サブケース | 月額コスト（1 アプリ）| 初期設定工数 | 追加運用工数 | Glue ETL 工数 |
|---|---:|---|---|---|
| **C-1: S3 のみ** | ~$5-10 | ゼロ | ゼロ | なし |
| **C-2: S3 + raw層 Catalog**（**推奨下限**）| **~$5-10**（Catalog はほぼ無料枠）| **数時間**（DDL 数本の登録）| **ほぼゼロ** | **なし** |
| **C-2+: S3 + raw + curated** | ~$10-15 | 1-2 日 | 数時間/月 | **軽微**（汎用処理のみ、用途独立で書ける）|
| **C-3: フルセット** | ~$15-25 | 1-2 日 | 数時間/月 | 大（用途駆動、後から追加）|

→ **C-2（S3 + raw層 Catalog）は追加コストほぼゼロ、設定工数も数時間、Glue ETL 不要**で、将来の移行工数を **18-36 ヶ月 → 6-12 ヶ月に短縮** できます。**強く推奨します**。

### 2.5.4 判断フロー

```
Step 1: 各アプリで SQL 分析（Athena / QuickSight / SageMaker）を使うか?
    │
    ├─ 使わない（S3 は純粋な保管のみ）
    │   ├─ 将来も横断分析の必要性はゼロか?
    │   │    ├─ YES → C-1: S3 のみ で OK
    │   │    └─ NO  → C-2: 最低限 raw層 Catalog は用意しておく
    │   │
    │   └─ アーカイブ・監査ログ用途で長期保管のみ
    │        → C-1: S3 のみ で OK
    │
    └─ 使う（Athena / QuickSight で SQL 分析する）
         → Step 2 へ

Step 2: Athena での使い方（クエリ内容、ダッシュ設計）は決まっているか?
    │
    ├─ 決まっていない or 検討中
    │   → C-2: S3 + raw層 Catalog のみ（Glue ETL は不要）
    │   → 用途が決まってから Glue ETL を追加すれば良い
    │
    └─ 決まっている（用途明確）
         → C-2+ or C-3: curated / analytics 層まで作る
             └─ 本格運用したい場合 → C-3 に格上げ検討
```

**重要**: Step 2 で「決まっていない」場合、**わざわざ curated / analytics 層を先に作らない**でください。用途未定の変換は将来やり直しになります。

### 2.5.5 C-1 が本当に成立する限定的なケース

以下**全て**に該当する場合のみ C-1（S3 のみ）で成立:

- [ ] 各アプリで **SQL 分析を一切しない**（Athena / QuickSight / SageMaker を使わない）
- [ ] 分析は各アプリの**管理画面**や**既存 BI ツール**（Aurora 直接接続 等）で完結
- [ ] S3 は完全な**アーカイブ・監査ログ**用途
- [ ] **将来的にも横断分析の必要性がゼロ**

→ 実際には稀なケース。多くの場合は **C-2 が実質的な下限** になります。

### 2.5.6 C-2「S3 + raw層 Catalog」の具体的な最小構成

C-2 で用意すべき最小限のもの（**Glue ETL は含まれません**）:

| 項目 | 内容 | 例 |
|---|---|---|
| Glue Database | データベース単位 | `expense_app_raw` |
| Glue Table | テーブル定義（スキーマ・パーティション情報）| `expenses`, `users` 等 |
| S3 LOCATION | Glue Table に紐付ける S3 パス | `s3://acme-expense-raw-prd/expenses/` |
| パーティション定義 | Partition Projection 推奨（[account-architecture-analysis.md §4.5.2](account-architecture-analysis.md) 参照）| `tenant_id / year / month / day` |
| Glue Crawler | **不要**（Partition Projection なら手動 DDL で十分）| — |
| **Glue ETL** | **不要**（用途が決まってから追加）| — |

**Athena で 1 回だけ実行する DDL 例**（raw 層に生データがある前提、変換なし）:

```sql
-- raw 層の生データに対する目録登録のみ、変換なし
CREATE EXTERNAL TABLE expense_app_raw.expenses (
    expense_id STRING,
    amount BIGINT,
    submitter_id STRING,
    submitted_at TIMESTAMP,
    status STRING
)
PARTITIONED BY (
    tenant_id STRING,
    year INT,
    month INT,
    day INT
)
STORED AS PARQUET
LOCATION 's3://acme-expense-raw-prd/expenses/'
TBLPROPERTIES (
    'projection.enabled' = 'true',
    'projection.tenant_id.type' = 'enum',
    'projection.tenant_id.values' = 'T-001,T-002,...',
    'projection.year.type' = 'integer',
    'projection.year.range' = '2026,2030',
    'projection.month.type' = 'integer',
    'projection.month.range' = '1,12',
    'projection.day.type' = 'integer',
    'projection.day.range' = '1,31',
    'storage.location.template' = 's3://acme-expense-raw-prd/expenses/tenant_id=${tenant_id}/year=${year}/month=${month}/day=${day}/'
);
```

→ **この DDL を各テーブル分（5-10 個）実行するだけ**で C-2 の設定完了（数時間の作業）。
→ **Glue ETL ジョブは 1 つも作らない**。用途未定でも設定可能な理由がこれ。
→ 用途が決まったら、後から Glue ETL で curated / analytics 層を追加すれば良い。

### 2.5.6b Parquet 形式に「変換」する部分の扱い

「生データが JSON なのに、なぜ Parquet で保存できるのか?」という疑問について:

| データソース | Parquet 変換の実施方法 | Glue ETL 必要? |
|---|---|:---:|
| Kinesis Data Firehose 経由 | **Firehose の Format Conversion 機能**（設定で有効化するだけ）| **不要** |
| DMS 経由（Aurora → S3）| **DMS の Output Format 設定**で Parquet 指定 | **不要** |
| Aurora Export to S3 | **Aurora Export の設定**で Parquet 指定 | **不要** |
| Lambda 経由（小規模）| Lambda 内で pandas / pyarrow で Parquet 変換 | **不要**（Lambda で完結）|
| 既存 CSV ファイルの一括変換 | Glue ETL または Athena CTAS | ⭕ 一時的に必要 |

→ **通常の取込パイプライン（Firehose / DMS / Aurora Export）は Glue ETL なしで Parquet 化できる**。
→ Glue ETL が本当に必要になるのは **業務ロジックを含む変換**（PII マスク、集計、結合等）だけ。

### 2.5.7 サブケース別の将来移行工数の実態

**C → A案 / B案 に移行するときの各サブケースの工数内訳**:

| 移行ステップ | C-1（S3 のみ）| C-2（S3 + Glue Catalog）| C-3（S3 + Glue フル）|
|---|---|---|---|
| データの発見・棚卸し | 3-6 ヶ月（何がどこにあるか調査から）| 数日（Catalog を見るだけ）| 数時間（自動化済）|
| スキーマの推論・定義 | 3-6 ヶ月（データを見て推測）| 不要（Catalog に登録済）| 不要 |
| パーティション設計 | 3-6 ヶ月（データを見て決定）| 不要（既に定義済）| 不要 |
| メタデータ登録 | 3-6 ヶ月（一から作成）| 不要 | 不要 |
| Federation / Replication 設定 | 1-2 ヶ月 | 1-2 ヶ月 | 1-2 ヶ月 |
| クロスアカウント認可設定 | 1 ヶ月 | 1 ヶ月 | 1 ヶ月 |
| データ整合性確認 | 3-6 ヶ月 | 1 ヶ月 | 数週間 |
| **合計工数（10 アプリ）** | **18-36 ヶ月** | **3-6 ヶ月** | **2-4 ヶ月** |

→ **C-1 と C-2 の差は 5-9 倍**、**数億円レベルの差**になり得ます。

### 2.5.8 推奨事項（この FAQ のまとめ）

| 状況 | 推奨サブケース |
|---|---|
| 各アプリで SQL 分析を使う予定がある（Athena / QuickSight 等）| **C-2: S3 + raw層 Catalog** |
| Athena 使途がまだ決まっていない | **C-2: raw層 Catalog のみ、Glue ETL は不要**（後から追加可）|
| Athena 使途が決まっていて、汎用クリーニング（PII マスク等）も欲しい | C-2+: raw + curated 層 |
| 各アプリで自動化・品質管理まで本格運用したい | C-3: フルセット（analytics 層まで）|
| 純粋なアーカイブ用途、SQL 分析なし | C-1: S3 のみ で OK |
| 将来の統合可能性を確保したい | **C-2 以上を推奨** |
| コストと工数を最小化しつつ将来の柔軟性も維持 | **C-2 が最適解** |

**→ 特別な理由がない限り、本補足資料は C-2 以上を前提として §3 以降の標準を提示しています。**

### 2.5.9 全体まとめ: 「Athena 使途未定でも今やること / 後回しにすること」

**今やる（Athena 使途未定でも実施すべき）**:

- [x] S3 バケット命名・レイアウト設計
- [x] Parquet + Snappy 形式で保管（Firehose / DMS / Aurora Export の設定で対応可、Glue ETL 不要）
- [x] パーティション: `tenant_id / year / month / day`
- [x] **Glue Data Catalog に raw 層のテーブル登録**（DDL 数本、数時間、Glue ETL 不要）
- [x] IAM / KMS / CloudTrail の設定
- [x] 機密度分類・PII フラグの付与（メタデータ）

**後で良い（Athena 使途が見えてから）**:

- [ ] Glue ETL ジョブの作成（curated 層 / analytics 層）
- [ ] 用途別集計テーブル・非正規化テーブル
- [ ] Glue Data Quality ルール
- [ ] Glue Crawler の自動化
- [ ] Athena Workgroup の設計・スキャン量上限

**この整理により、「Athena 使途未定でも Glue Catalog 登録は今やる意味がある」ことがご理解いただけると思います。**

## §3 メタデータ管理の標準（データ目録）

### 3.1 標準 5: Glue Catalog（データ目録）の登録

**推奨**: 各アプリで **AWS Glue Data Catalog にテーブルを登録** する

**理由**:
- Glue Catalog は AWS 標準のメタデータストア（テーブル定義・スキーマ・パーティション情報）
- 将来の中央から Federation（横断参照）で機械的に読み込める
- 各アプリの Athena もこの Catalog を参照するため、**「今すぐ使う理由」も十分**

**登録すべき情報**:
- テーブル名
- 列定義（列名・型）
- パーティション列
- S3 の場所（LOCATION）
- ファイル形式（Parquet）
- 圧縮方式（Snappy）

**避けるべき**:
- Excel やスプレッドシートで台帳管理（機械的に読めない）
- テーブル定義を SQL ファイルにしか書かない
- Catalog を使わず、都度 CREATE TABLE で運用

### 3.2 標準 6: スキーマ共通化の 3 段階アプローチ

**現時点で共通化する必要はない**が、以下の 3 段階で徐々に揃える:

| 段階 | 内容 | 現時点で必要 |
|---|---|:---:|
| **段階 A**: **命名規則の共通化** | 顧客企業 ID は `tenant_id`、日付は `event_date` 等、**列名を統一** | ⭕ 今すぐ |
| **段階 B**: **型の共通化** | `tenant_id` は VARCHAR(20)、`event_date` は DATE 等、**型を統一** | ⭕ 今すぐ |
| **段階 C**: **業務データモデルの共通化** | 「顧客」「取引」「イベント」等の共通エンティティを設計 | △ Phase 2 |

**理由**:
- 段階 A/B は追加コストほぼゼロ、統合時に**変換不要**
- 段階 C は業務レベルの合意が必要なので後回しでも良い

### 3.3 標準 7: 機密度・PII 情報の分類ラベル

**推奨**: 各テーブル・列に **機密度分類** を付与:

| 分類 | 例 |
|---|---|
| Public | 公開情報（会社名、業種等）|
| Internal | 社内利用のみ（内部業績等）|
| Confidential | 機密（財務情報、顧客契約情報等）|
| Restricted | 高機密（個人情報、認証情報等）|
| PII: Yes/No | 個人情報を含むか |

**理由**:
- 将来の中央からのアクセス制御（Lake Formation LF-Tag 等）で必須
- GDPR/APPI 対応にも直結
- **各アプリ内でも監査で必要になる**

**実装**: Glue Catalog のテーブル・列に AWS Lake Formation LF-Tag として付与、または列コメントに記載

## §4 アクセス制御・セキュリティの標準

### 4.1 標準 8: IAM Role でアクセス制御（API キーの禁止）

**推奨**: S3 / Glue へのアクセスは **IAM Role で制御**、API キー直書きは禁止

**理由**:
- 将来の中央からのアクセスも IAM Role Assume で機械的に付与できる
- 監査ログ（CloudTrail）で誰がアクセスしたか追跡可能
- **各アプリ内でもセキュリティのベストプラクティス**

### 4.2 標準 9: KMS 暗号化の適用

**推奨**: 分析用 S3 バケットは **AWS KMS で暗号化**（SSE-KMS）

**理由**:
- 将来、中央から Cross-account でアクセスするとき、KMS 鍵ポリシーの調整だけで許可できる
- **各アプリ内でもコンプライアンス的に必須**

### 4.3 標準 10: 監査ログ（CloudTrail Data Events）の有効化

**推奨**: 分析用 S3 バケットの **CloudTrail Data Events** を有効化

**理由**:
- 誰が何を読んだかの記録は将来の統合検討時にも必要
- **各アプリ内でもコンプライアンス的に必須**

## §5 将来移行のロードマップ

### 5.1 ロードマップ全体像

```
Year 1-3: C案 個別型 で運用（標準を守りながら）
    │
    │ 【契機発生】ビジネス環境変化 or SaaS 数増加
    │
    ├─ 選択肢 1: A案（分散型）への移行
    │  Phase 1: 中央基盤構築（3 ヶ月）
    │  Phase 2: 1-2 アプリで Federation 連携試行（2 ヶ月）
    │  Phase 3: 全アプリ順次 Federation（3-6 ヶ月）
    │  合計: 6-12 ヶ月
    │
    └─ 選択肢 2: B案（中央集約型）への移行
       Phase 1: 中央基盤構築（3 ヶ月）
       Phase 2: 1-2 アプリで S3 Replication 試行（2 ヶ月）
       Phase 3: 全アプリ順次 Replication + 中央 ETL 移行（6-12 ヶ月）
       合計: 12-18 ヶ月
```

### 5.2 移行時の変換工数（10 アプリの場合）

| 変換項目 | 標準を守っている場合 | 標準を守っていない場合 |
|---|---|---|
| S3 レイアウト変換 | 不要 | 全データ再配置、テナント別データ分解 |
| ファイル形式変換 | 不要 | CSV → Parquet 変換、圧縮適用 |
| メタデータ整備 | 既に Glue Catalog に登録済 | 台帳から手動登録、スキーマ推論 |
| スキーマ統一 | 命名規則が揃っているため軽微 | 全列名の突合・変換テーブル作成 |
| アクセス制御設定 | IAM Role の Cross-account 設定のみ | 認証方式の全面書換、API キー廃止 |
| **移行工数（合計、1 アプリ）** | **1-2 週間** | **4-8 週間** |
| **10 アプリ合計** | **3-6 ヶ月** | **12-24 ヶ月** |

### 5.3 移行判断のトリガー

以下のいずれかが発生したら、**A案 / B案 への移行検討を開始**:

| トリガー | 検討タイミング |
|---|---|
| SaaS 製品数が 5 個以上になった | 各年度末 |
| 経営陣から全社 KPI ダッシュボード要望が出た | 即時 |
| 大口顧客から複数 SaaS 横断要望が来た | 即時 |
| 業界規制（GDPR/APPI 等）で一元管理が必要に | 即時 |
| 事業戦略変更・組織再編 | 都度 |

## §6 各アプリ側への推奨事項（チェックリスト）

現時点で C案 個別型 を採用する場合、各アプリ側で以下を**必ず実施**することを推奨:

### 6.1 必須項目（追加コストほぼゼロ）

- [ ] 分析用データは Amazon S3 に保管する
- [ ] バケット命名規則を統一（`<会社略称>-<アプリ名>-<データ層>-<環境>`）
- [ ] Medallion 3 層（raw / curated / analytics）で構造化
- [ ] Parquet + Snappy 圧縮で保管
- [ ] パーティション設計: `tenant_id / year / month / day`
- [ ] AWS Glue Data Catalog にテーブル登録
- [ ] 列名・型を業界標準に揃える（`tenant_id` = VARCHAR(20) 等）
- [ ] 機密度分類（Public / Internal / Confidential / Restricted）と PII: Yes/No を各列に付与
- [ ] IAM Role でアクセス制御（API キー直書き禁止）
- [ ] KMS 暗号化を適用
- [ ] CloudTrail Data Events を有効化

### 6.2 推奨項目（工数少・効果大）

- [ ] AWS Glue Crawler でスキーマ自動検出
- [ ] AWS Glue Data Quality でデータ品質チェック
- [ ] S3 Object Lock で改竄防止（監査対応）
- [ ] S3 Lifecycle で古いデータを Glacier に移動（コスト最適化）

### 6.3 将来対応項目（今は不要、将来必要）

- [ ] 業務データモデルの共通化（顧客・取引・イベント等の共通エンティティ）
- [ ] Apache Iceberg テーブル形式への移行検討（Time Travel、ACID 対応）
- [ ] Aurora Zero-ETL for S3 の導入検討

## §7 追加コスト vs 将来メリット の総括

### 7.1 標準を守る追加コスト

| 項目 | 追加コスト |
|---|---|
| S3 保管形式（Parquet + Snappy）| ゼロ（むしろストレージ 75% 削減）|
| バケット命名 | ゼロ（命名するだけ）|
| Medallion 3 層構造 | ゼロ（設計時に決めるだけ）|
| パーティション設計 | ゼロ（設計時に決めるだけ）|
| Glue Catalog 登録 | ゼロ（無料枠内で足りる）|
| 列名・型の統一 | ゼロ（設計時に決めるだけ）|
| 機密度分類 | ゼロ（付与するだけ）|
| IAM Role / KMS / CloudTrail | ゼロ（セキュリティで元々必要）|
| **総追加コスト** | **ほぼゼロ**、むしろ **各アプリ内でも品質・セキュリティ向上** |

### 7.2 将来の統合メリット

| 項目 | メリット |
|---|---|
| C → A案 移行工数 | 12-24 ヶ月 → **3-6 ヶ月**（3-4 倍短縮） |
| C → B案 移行工数 | 18-36 ヶ月 → **6-12 ヶ月**（3 倍短縮）|
| 10 アプリの移行コスト | $1-3M → **$300-500K**（数倍削減）|
| 統合後の運用品質 | 標準化されているため運用工数 30-50% 削減 |
| セキュリティ・監査対応 | 統合時に個別対応不要、機械的に反映 |

### 7.3 結論

**「集約しない（C案）」を選んでも、以下の標準を守れば将来の統合可能性を残せる**:

1. S3 + Parquet + Snappy + パーティション（`tenant_id / year / month / day`）
2. Glue Catalog 登録
3. 列名・型の統一
4. 機密度分類の付与
5. IAM Role / KMS / CloudTrail の徹底

**これらは追加コストほぼゼロ、むしろ各アプリ内でも品質向上します**。ヒアリング結果を踏まえて C案 が選ばれた場合でも、**各アプリ側の実装ガイドラインとしてこれらの標準を提示することを推奨**します。
